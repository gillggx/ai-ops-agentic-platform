從單一 Agent 到治理化的 Multi-Agent 平台

這一輪把平台重構成四件事：多 Agent 的明確分工、知識與規則的來源治理、記憶的三級信任階梯、以及四語系支援。本文著重「做了什麼能力、為什麼這樣設計」，運維細節隨附。

Section 1

1架構總覽

四個服務、各司其職、邊界清楚。核心設計原則：單一 DB 擁有者（只有 Java 碰 PostgreSQL），Agent 全部住 sidecar，前端只做 UI 與 proxy。

:8000 aiops-app Next.js standalone。UI 渲染 + /api/ proxy。不直接連 sidecar/simulator/DB。 Frontend
:8002 aiops-java-api Spring Boot，唯一 DB 擁有者。所有 PostgreSQL 讀寫、JWT auth、business CRUD、pipeline/skill/alarm registry、治理簽核。 Java · DB
:8050 python_ai_sidecar LangGraph + Pipeline Executor。所有 Agent 住這裡（chat orchestrator、Glass Box builder、Supervisor）。透過 Java client 對話，永不直連 DB。 Agents
:8012 ontology-simulator 純資料服務（MES 後端），API 與 production ontology 相同。不知道 Agent 的存在。 Data Source
:8003 aiops-java-scheduler 排程觸發（Auto-Patrol、記憶生命週期 janitor）。 Scheduler
資料流原則：前端 → Java（一切業務） → sidecar（Agent 推理） → simulator（取資料）。sidecar 需要任何 DB 資料都經 Java client，不存在 sidecar → PostgreSQL 的直連。這讓 DB schema 演進只需改一處，也讓 Agent 層可獨立擴縮。
DB：PostgreSQL + pgvector（knowledge/memory embedding）· MongoDB（simulator） 部署：EC2 systemd · deploy/update.sh（前端）· java-update.sh（Java+sidecar）
Section 2

2Agents 分工

執行組織是 code 資產：改動走 git → deploy → gate，一般 user 不直接編輯。每個 Agent 只做窄任務，「下一步做什麼」由 graph 決定，不塞進 prompt。

Agent職責關鍵設計
Planner把需求拆成 phase 計畫（goal_plan / replan）block-agnostic，只寫 intent 不綁 block
Builder選 block、填參數、組 DAG（agentic_phase_loop）每 phase 有 round 預算；空回應守門避免鬼打牆
Repair卡關時三路對齊診斷、修 canvas需求 × canvas × 結構化拒因；明確移除壞節點
Verifier結構 + 實測驗收零 LLM，純確定性 — 拒因可單測、可信
Chat orchestrator對話路由（v2, LangGraph）intent_classifier + intent_completeness 皆 graph node
Supervisor平台 SRE + 知識策展人（見第 6 節）只有提案權，落地一律經人簽核
核心原則 — flow 寫 graph，不寫 prompt：任何「下一步該做什麼／呼叫哪個 tool」的決策禁止塞進 LLM system prompt。理由：LLM 的自由意志會違抗 prompt 規則（已多次證實）；graph node 是 pure function、可單測；出錯時 graph 知道是哪個 node fail，prompt-flow 只能猜「LLM 又走偏了」。LLM 在每個 node 內只做思考型工作（分類／抽參／寫答），不決定下一步。

Verifier 為什麼零 LLM

若驗收也用 LLM 判斷 → 不穩定、無法單測、每個 case 都可能飄
→
結構檢查（param schema / port 相容）+ 實測（跑一次看 rows）都是確定性程式碼 → 拒因結構化、可回饋給 Builder 下一 round 精準修正
Section 3

3知識與規則

工作手冊是 data 資產：改動走 UI → DB，維護者是人（PE / IT_ADMIN）與受治理的 Agent 寫入。核心原則：文件是 LLM 的唯一來源。

類別用途維護使用者
Directives每次回覆必掛的硬規則IT_ADMINchat agent（every turn）
Knowledge領域知識，RAG 按相關性取用（plan / execute 分層 + always_on）人 + Agent 寫入（經治理）Planner、Builder
Lexicon行話 → 標準詞改寫PEchat 前處理
Examples回答風格示範（few-shot）PEchat agent
Block docsblock 的唯一文件來源人終審Builder LLM、Block Advisor
MCP description外部資料源的唯一文件來源IT_ADMINorchestrator、生成器
禁止 hardcode 文件到 prompt：LLM 需要了解某個 block／MCP 時，一律從 DB 動態讀 description + input_schema。理由：若 block 行為改了但 prompt 的 hardcode 沒跟著改，LLM 會產生錯誤的 code。單一來源（DB）才能保證前端、Agent、文件抽屜三邊一致。
禁止把 case-specific rule 塞進 prompt：某個 case 失敗 → 不是「再加一條規則」，而是問根因原則。每條 case rule 六個月後都會被新變形繞過。正確做法：抽象成一句原則寫 prompt、或移到 graph node 做確定性檢查、或改 schema。
Section 4

4記憶重構 — 三級信任階梯

Agent 在戰場上留下三種筆記，可信度遞減、生效門檻遞增。訊號來源決定生效速度。這是這輪最核心的架構決策。

W1
偏好（Planner 寫） — user 親手修改 plan（例：SPC 圖要附 UCL/LCL）。全系統可信度最高的訊號，因為是 user 本人的動作。立即生效 個人 scope，/me 我的偏好 可自助管理。
W2
文件便利貼（Builder 寫） — agent 踩坑後的自述（例：block_unnest 的 limit 型別易誤）。改的是全域共用文件，錯了污染所有人。進待審佇列 agent 不准直接改 block docs。
W3
修正（Repair 寫） — agent 從一次修復推廣的通則，可能只是巧合。draft，不參與檢索 等同這筆知識還不存在，需 Supervisor 找旁證轉正。

生命週期與索引（V75 新增）

狀態機

draft → active → stale → archived

  • draft 逾 30 天未轉正 → 自動 archived
  • episodic 逾 90 天 → stale
  • preference 零召回 180 天 → 提 PRUNE
  • domain/procedure 年審（review_at）

確定性 janitor 每日跑（Java scheduler），非策展判斷。

索引 + 檢索權重

subject 索引：每筆記憶掛 subject_kind（block/tool/skill/request_class）— 讓「按 block 聚合」查案成為可能。

權重檢索（flag，預設 OFF）：cosine × priority × 180天半衰期 + 每類名額（preference ≤1、episodic ≤1）防單一類洗版。

關鍵擔憂「久了只是一堆 memory」→ 由代謝（PRUNE/MERGE）+ 名額回答。

Section 5

5多語系支援

四語系：繁中 / 簡中 / English / 日本語。核心決策：介面殼翻譯、資料不翻（source is truth）、對話交給 LLM。

介面殼

next-intl v4，cookie-based 無 URL routing（內部工具不需 SEO）。14 個 namespace，繁中為源語言。GLOSSARY 術語約束（OOC/Cpk 保留英文）。check-i18n-literals 防回歸。

資料值

tool_id、chart、DB 的 block/skill 描述、記憶內容一律原貌。source is truth — 生成物保持生成時語言，不回溯翻譯。

LLM 對話

回覆跟隨 user 語系（locale_directive 注入 prompt）。繁中回空字串保持 prompt byte-identical（不影響基線/cache）。日文用です・ます体。

設計優勢：wire 上送「code + 結構化參數」、顯示層才翻譯（同 event-wrapper 原則）。因此 Java / sidecar 完全不用知道有幾種語言 — 加一個語系只動前端 catalog。users.locale 存跨裝置偏好，cookie 是本機快取。
Section 6

6治理閉環 — Supervisor

Supervisor 是平台的 SRE 兼知識策展人：監督執行體系的健康與知識體系的品質，產出有證據的提案。硬規則：只有提案權，落地一律經人簽核。

三個監督面

S1 執行健康

build 成功率、rounds、拒因分布、退步偵測 → 診斷 + 調參提案

S2 知識品質

召回命中、W1–W3 佇列、零召回事件 → PRUNE/PROMOTE/MERGE/DOC_REVISE

S3 成本資源

花費、cache、provider 品質（空回應率）→ 異常警報 + 設定調整提案

查案（Case Lesson Learned）— 這輪的旗艦能力

Supervisor 自己翻失敗紀錄找根因：把「開發者人肉盯 trace」自動化。防垃圾記憶是硬規則：

  • 單一 case 熱點 → 什麼都不留（不值得任何記憶）
  • 知識提案必須 ≥3 個不同 request 的同型失敗 + 泛化聲明
  • 修文件優先於加記憶（改一次蓋住所有未來 case）
  • 成本閘：每次 ≤3 個 LLM 深挖、去重在 LLM 之前、選案/彙整/回寫零 LLM → 一次巡檢約 7–17k tokens ≈ 幾美分，比一個 user build 還省

簽核路由 — 信任階梯

來源誰批為什麼
ON_DUTY 現場人員（教它/回報）PE驗證現場經驗的領域正確性
Agent 提的（策展、增量/可復原）Supervisor 自動核agent 提的由 Supervisor 同意即可；全程 audit
Supervisor 查出來的（forensics）+ 設定/PRUNE 刪除IT_ADMIN平台級決策 + 刪除永遠留人

提案的生命週期

提案 → 簽核 → 落地 → 成效回寫

每個提案是案情四段（發生了什麼 / 觀察到的問題 / 影響對象 / 提議）+ 證據鏈（▣ 系統事實 · △ 自述 · ◆ 記憶，可點回原始 build 紀錄）。落地滿 7 天 Supervisor 自動回寫成效（「拒因 X n→m」）。手動觸發時有阻擋式進度罩（共 N 筆、檢查到第 X 筆），activity-id 防重複，舊提案自動 supersede。

Section 7

7運維速查

給值班 DevOps 的最小操作集。

部署
  • deploy/update.sh — 前端 + simulator（自動 restart aiops-app、ontology-simulator）
  • deploy/java-update.sh — Java + sidecar（rebuild jar + venv，restart 兩者）
  • ⚠ update.sh 不重啟 sidecar/Java；改 sidecar 或 Java 要跑 java-update.sh
DB migration
  • Flyway 在 prod 停用 — 新 V**.sql 要 SSH EC2 手動 psql -f
  • System MCP 每次啟動自動 sync（canonical 在 main.py）
  • pgvector 欄位禁用 JPA save，走 native CAST(:vec AS vector)
排程
  • 記憶 janitor：Java scheduler，每日 03:20（生命週期歸檔）
  • Supervisor forensics：crontab 每日 03:40（/var/log/supervisor-forensics.log）
  • Auto-Patrol：event/schedule 觸發
Stack 版本（須跨檔一致）
  • Java Temurin 17 · Spring Boot 3.5.14 · Maven
  • Python 3.11 · Node.js 20.18
  • Frontend output: "standalone"
