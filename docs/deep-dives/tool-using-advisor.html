Tool-Using Doc Q&A Agent (advisor_v2)

v6.2 — Anthropic 標準 tool-use loop。EXPLAIN / COMPARE / RECOMMEND 三 intent 走這。LLM 自己決定要 query 哪個 block doc, 不再 pre-fetch 整 catalog。

動機

舊 Advisor 是 graph-based: classifier → extractor → fetcher (pre-fetch java.list_blocks() 全部 55 block) → synthesize. 兩個問題:

  • 看不到 admin-edited Markdown: fetcher 走 /internal/blocks 拿 block_definitions, 跟 block_docs table 是分開的。Admin 在 /admin/block-docs 編的 rich Markdown body 永遠進不了 advisor 答案。
  • Scale 不好: pre-fetch 全部, 換新 question 換新 fetcher node, 不靈活。

新做法跟 memory project_rag_for_llm_lookups.md 方向一致: 「棄死板 push 全清單, 改 LLM 主動 query」.

架構

/internal/agent/build (user_message) │ ▼ classify_advisor_intent (LLM, 5-bucket) │ ├─ BUILD ────── stream_graph_build (separate doc) ├─ EXPLAIN ────┐ ├─ COMPARE ────┤ ├─ RECOMMEND ──┘── stream_doc_qa_agent (THIS DOC) │ ├─ KNOWLEDGE ────── legacy advisor (pure LLM, no fetch) └─ AMBIGUOUS ────── legacy advisor (clarify message) (EXPLAIN/COMPARE/RECOMMEND path:) │ ▼ stream_doc_qa_agent (Anthropic tool-use loop) │ ├── round 1: LLM emit tool_use(s) │ │ │ ▼ │ dispatch each: list_blocks / search_blocks_by_keyword / inspect_block_doc │ │ │ ▼ │ append tool_result to messages │ │ │ ▼ (next round) │ └── round N: LLM emit text only (no tool_use) │ ▼ final markdown answer │ ▼ emit advisor_answer SSE event │ ▼ emit done SSE event Round cap: 5 Max tokens per round: 2048

The 3 tools

ToolInputReturnsWhen to use
list_blocks category? (string) 1-line per block: name + category + headline User asks about category ("chart blocks 有哪些"). Returns ~55 blocks if no filter.
search_blocks_by_keyword keyword (string) top-6 substring matches across name+description+tags User describes intent ("count OOC", "detect drift"). Get candidates without inspecting full doc.
inspect_block_doc block_id (string) Full Markdown body (from block_docs DB; falls back to seed description) Once candidate identified. Returns When-to-invoke / 不適用情境 / Inputs / Outputs / Parameters / Examples.

System prompt

你是 pipeline-builder 的 block 使用 Q&A 助手。User 問 block 怎麼用 /
哪個 block 適合 / 兩個 block 差在哪。你**主動 query** 工具拿 docs 再回答,
不要憑記憶。

決策原則:
  1. user 提到 specific block name → 直接 inspect_block_doc 該 block
  2. user 講意圖 / keyword → search_blocks_by_keyword 找候選 → 再 inspect 1-2 個
  3. 比較問題 → inspect_block_doc 各別拿 doc, 並列回答
  4. 推薦問題 → search → inspect top 2-3 → 比較後給推薦
  5. 拿到 doc 後**直接回答**, 不要再無謂 inspect

回答格式 (Markdown):
  - 簡明 2-5 段
  - 若涉及 params 用 table
  - 若有 example chain 用 code block
  - 結尾不要客套, user 是 engineer

Graph-heavy 偏好的調和

CLAUDE.md memory 講「flow control 寫 graph 不寫 prompt」。Build agent 走 LangGraph 是這原則。但 doc Q&A 反而用 tool-loop, 怎麼協調?

Graph-heavy 規則的 context 是 BUILD task — flow control 重要、有 state mutation、出錯成本高。Doc Q&A 不一樣:
• 純 read, 沒 mutation
• 每個 question 該查什麼差很多 — graph 寫死 fetcher 反而不好
• 工具受控 (3 個 read-only) + round cap (5)
所以 Q&A 走 tool-loop 是對的, 跟 build 走 graph 不衝突。

跟 build agent 用同套 tools 嗎?

Build agent (BuilderToolset in tools.py) 有 ~25 個 tools, 大部分是 canvas mutation (add_node, connect, set_param, ...)。Q&A 不需要這些, 只需 read-only doc tools。所以 advisor_v2 自定義 3 個 minimal tools, 不繼承 BuilderToolset。

不過 inspect_block_doc 的實作走 java_client.get_block_doc(block_id, version), 跟 build agent 走的 _resolve_block_description 是同一個 DB 來源, 所以兩邊看到的 doc 一致。
