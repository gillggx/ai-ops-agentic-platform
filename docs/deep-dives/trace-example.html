Trace Example — EQP-08 boss case

真實 build trace 走過一輪 LangGraph + agent-driven verify + multi-block phase chains。一次 finished, 5 phases, 7 nodes, 16 LLM calls。

The prompt

查詢 EQP-08 最近 7 天內最後一次 OOC 事件，確認該次是否有超過 2 張 SPC 同時 OOC，並顯示該次 OOC 時所有 SPC charts 的狀況

需求拆解 (planner 抽出 5 phase):

Phaseexpected意圖
p1raw_data取得 EQP-08 過去 7 天的歷史資料
p2transform找出最後一次 OOC 事件的時刻 (sort desc + take first)
p3verdict判定該次同時 OOC 的圖表數是否 ≥ 2 (unnest + filter + groupby + threshold)
p4table展示該時刻所有 OOC 圖表 (data_view)
p5chart視覺化 (bar_chart)

Final canvas

n1 process_history (EQP-08, 7d, limit=200, nested=true) │ 200 events × 12 spc_charts each = 2400 unnested rows ▼ n2 block_find (spc_status=OOC, order_by=eventTime desc, take=first) │ → 1 row (最後一次 OOC 事件, 包含完整 nested spc_charts) ▼ n3 block_unnest (column=spc_charts) │ → 12 rows (該事件的 12 個 chart 各自展開成 row) ▼ n4 block_filter (column=status, operator===, value=OOC) │ → N rows (該事件中 status=OOC 的圖表, 答 user "有幾張 OOC") ├──────────────────────────┐──────────────────────────┐ │ │ │ ▼ ▼ ▼ n5 block_step_check n6 block_data_view n7 block_bar_chart (count >= 2) (table view) (x=name, y=value/ucl/lcl verdict table highlight=OOC) chart

Phase-by-phase 解讀

p1 raw_data — 5 rounds

r1   inspect_block_doc block_process_history
r2   commit_pick block_process_history
r3   add_node process_history (tool_id='EQP-08', time_range='7d', limit=200, nested=true)
                              → preview success, 200 rows
r4   inspect_node_output n1   → 確認 columns
r5   run_verifier             → ADVANCED

Agent 主動 inspect 確認資料 shape 再 run_verifier。標準 fast path。

p2 transform — 6 rounds

r1   inspect_node_output n1 (再確認)
r2   inspect_block_doc block_find  ← 認識「找最後一次」用哪個 block
r3   commit_pick block_find
r4   add_node block_find (column='spc_status', operator='==', value='OOC',
                          order_by='eventTime', order_dir='desc', take='first')
                                   → preview success, 1 row
r5   connect n1.data → n2.data
r6   run_verifier                  → ADVANCED

User 「最後一次」→ agent 翻譯成 sort desc + take first。一個 block 完成。

p3 verdict — 14 rounds multi-block chain

r1-2  inspect_block_doc block_unnest, block_filter
r3    commit_pick block_unnest
r4    add_node block_unnest (column='spc_charts')  → n3, 12 rows
r5    connect n2.data → n3.data
r6    inspect_node_output n3  → 確認展開後 columns
r7    commit_pick block_filter  ← 鏈第 2 個 block (in tune subphase)
r8    add_node block_filter (column='status', operator='==', value='OOC')  → n4
r9    connect n3.data → n4.data
r10   inspect_node_output n4
r11   commit_pick block_step_check  ← 鏈第 3 個 block
r12   add_node block_step_check (aggregate='count', operator='>=', threshold=2)  → n5
r13   connect n4.data → n5.data
r14   run_verifier                  → ADVANCED
重點觀察: p3 一個 phase 內鏈了 3 個 block (unnest → filter → step_check)。這要靠 v30.22 agent-driven verify — agent 自己決定何時 run_verifier, 不會在第一個 add_node 後就被擋住。Sub-phase machine 的 ("tune", "commit_pick"): "construct" transition 讓 agent 從 tune 直接接下個 block (v30.22 新加)。

p4 table — 4 rounds

r1   commit_pick block_data_view  ← agent 已從 p3 知道 data shape, 直接 pick
r2   add_node block_data_view (params={})
r3   connect n4.data → n6.data    ← fan-out 從 n4 (filter result) 另開一條
r4   run_verifier                 → ADVANCED

沒 inspect (agent 已熟悉 data_view), 直接 add + connect + verify。最快 path。

p5 chart — 8 rounds

r1-4  inspect_block_doc × 4 (找適合的 chart type)
r5    commit_pick block_bar_chart
r6    add_node block_bar_chart (x='name', y=['value','ucl','lcl'],
                                 highlight_field='status', highlight_eq='OOC')  → n7
r7    connect n4.data → n7.data   ← 第 3 個 fan-out from n4
r8    run_verifier                → ADVANCED

Chart phase agent 花較多 inspect 挑 block (4 round) — 18 種 chart block 選擇豐富。

Verifier 為何全過

PhaseTerminal blockChecks (covers gate OFF)結果
p1n1 process_historyvalidation_error: no, orphan: source so skipADVANCED
p2n2 block_findvalidation_error: no, orphan: has n1 upstreamADVANCED
p3n5 step_check (canvas latest after run_verifier)validation_error: no, orphan: has n4 upstreamADVANCED
p4n6 data_viewvalidation_error: no, orphan: has n4 upstreamADVANCED
p5n7 bar_chartvalidation_error: no, orphan: has n4 upstreamADVANCED

Covers gate OFF (v30.23 default), 所以 step_check covers=[scalar,verdict] 不需要嚴格對齊 p3.expected=verdict — 只看結構正不正常。

修法 (按貢獻排序):

  1. block_groupby_agg 命名對齊: agg_func/agg_column → aggregate/column (跟 step_check + pandas convention 一致)。Executor 雙接受 backward compat。
  2. v30.22 agent-driven verify: p3 multi-block chain 之前 verifier 加一個 block 立刻擋, 現在 agent 主動觸發才驗
  3. v30.23 covers gate off default: step_check covers=[scalar,verdict] 對 verdict phase OK, 不會 false-reject
  4. MAX_REACT_ROUNDS 8→16: multi-block phase 預算夠

如何自己跑這個 case

Via verify-build skill (CLI)

python3 .claude/skills/verify-build/run.py \
  --mode builder \
  --message "查詢 EQP-08 最近 7 天內最後一次 OOC 事件，確認該次是否有超過 2 張 SPC 同時 OOC，並顯示該次 OOC 時所有 SPC charts 的狀況" \
  --json-out /tmp/eqp08.json

Via Web UI

  1. 到 /admin/pipeline-builder/new
  2. chat 區輸入上面那段 prompt
  3. plan card 出現 → 按 Apply
  4. watch canvas 一個 block 一個 block 出現
  5. 結束後 → 到 /admin/build-traces 找最新 trace → 看 Summary tab
