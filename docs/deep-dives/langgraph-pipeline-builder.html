LangGraph Pipeline Builder

v30.23 — graph-controlled ReAct loop with sub-phase state machine + agent-driven verify trigger + deterministic verifier (covers gate off by default).

核心哲學

Flow control 寫在 graph，不寫在 prompt。LLM 在每個 node 只做窄任務 (classify / extract / write_params)。Phase transition / sub-phase routing / verify trigger 全 deterministic code，可單測、不會因 prompt drift 行為改變。

對應 memory feedback_flow_in_graph_not_prompt.md + feedback_graph_heavy_preference.md。

定義在 agentic_phase_loop.py。 _TOOLS_BY_SUBPHASE + _TRANSITIONS 兩個 dict 控制行為:

  add_node
  connect

Verifier (v30.20 + v30.23)

定義在 phase_verifier.py。Deterministic structural-only checks:

Check狀態說明
(A) covers gateOFF defaultblock 的 covers_output 包含 phase.expected。 BUILDER_VERIFIER_COVERS_GATE=1 才開。預設關閉因為實務上更多 case 是 false-positive reject
(B) validation_errorONblock executor 短路 / params 寫錯 → reject with error_message
(C) orphan checkONnon-source block 沒接 upstream edge → reject with "connect X"

之前 v30.x 有的全部砍掉, 移到 runtime concern:

  • LLM judge — 太常 over-reject 結構正確的 canvas
  • rows quality gate — 0 rows 是合法 filter 結果
  • deficit detection — 資料源天生少不該擋 build
  • empty_data routing — 同上
  • FF chain — 跟 covers 一起綁，default off 時自然停用

Agent-driven verify (v30.22)

之前 verifier 每個 mutating action 後自動跑 — 結果 agent 加 filter (transform) 到 verdict phase 立刻被擋, 沒機會接 step_check 完成 chain。

改成 router 看 state.v30_verify_now 決定:

def _route_after_phase_loop(state):
    if state.get("status") == "phase_revise_pending":
        return "phase_revise"
    if state.get("v30_verify_now"):
        return "phase_verifier"
    return "agentic_phase_loop"   # 回 loop, agent 繼續建

agentic_phase_loop 設 v30_verify_now = (tool_name in {"run_verifier", "phase_complete"}). Agent 自己宣告完成才驗。

System prompt 摘錄

住在 agentic_phase_loop.py 的 _SYSTEM constant。重點段落:

== 每 round 1 動作原則 ==
你每 round **只發一個 tool call**, 等系統執行完拿到結果 (auto-preview snapshot),
再進入下個 round 觀察新狀態決定下個動作。**不要連 emit 多個 add_node**。

== Phase 達成 — 你決定何時 verify (v30.22) ==
verifier **不再** 每個 action 後自動跑。你可以自由 add_node / connect /
set_param **多個 block** 建一條 chain (如 OOC 計數要 filter → step_check
兩 block)。當你覺得這個 phase 的 chain 建完整了, **主動呼叫
run_verifier()** 觸發 verifier 檢查。

verifier 跑時會做 deterministic 結構檢查 (不是 LLM judge):
  (a) validation_error: 鏈上任何 block executor 短路 / params 錯
  (b) orphan: 非 source block 沒接上游
  (c) covers gate: terminal 的 covers 是否含 phase.expected
      — **預設關閉**, phase.expected 退為 plan / prompt 內的「目標 hint」

== Param naming 嚴格規則 ==
add_node 的 params key 必須 **100% 一字不差** 從 inspect_block_doc
的 param_schema 抄過來。不要用同義詞替換。
