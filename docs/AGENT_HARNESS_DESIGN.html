1先看 Claude Code（我）自己怎麼「學」

關鍵事實:我不更新權重。我的「學習」全部外部化到 harness 的檔案 — 這跟你們的 agent 一樣(你們也不 fine-tune),所以這個範本可以直接搬。

我的「記憶」= 磁碟上的 memory/*.md(每檔一個 fact,frontmatter 分類 user/feedback/project/reference)+ CLAUDE.md(常駐憲章)+ skills(可重用程序)。 被糾正時,我寫一條 feedback 記憶(帶 Why + How to apply),下次靠 description 召回。 「學習」= 寫進一個分類好、可召回的記憶庫,不是改權重。你們的 Supervisor 要做的就是這件事。

6 個 memory class × 3 個正交軸。軸的部分你們 agent_knowledge 幾乎都有了 — 只要再加一個 class 維度、每個 agent 讀寫自己那片。

結論:不用重造 memory 基建。加一個 class 欄位到 agent_knowledge,搭配既有 scope/layer/priority,就是一套完整的分類記憶庫。

4兩種學習速度:Fast Path vs 蒸餾

Fast Path — in-session（即時)

Planner 收到 user feedback → 當下就寫一條 preference / correction(草稿) + 立刻重規劃。就像我在 session 中被糾正、馬上寫 memory 並改做法。特性:快、局部、未去重。

Slow Path — Supervisor 蒸餾（週期)

Supervisor 定期把大量 episodic traces 蒸餾成乾淨、去重、分類的 durable memory,並寫回。就是「反思/固化」。Supervisor = memory consolidator —— 把噪音經驗變成長期知識,還能發現「這是 doc 不齊」。

7幾個關鍵判斷

  1. Feedback intake 是唯一真正要新建的機制。其餘 9 個構件你們或多或少有了。這個機制 = 「user/Planner 的糾正 → 蒸餾成一條帶 why 的 correction 記憶」,就是我被糾正時做的事。沒有它,agent 每次都從零開始。
  2. Planner 同時是需求擁有者與裁判。它出需求給 Builder、又判 Builder 結果對錯 —— 這個「裁判」動作本身要留成 episodic,因為它是最有價值的訓練訊號(Supervisor 蒸餾的主料)。
  3. Repair 的三路對齊(原意圖 vs 現結果 vs feedback)就是它的 harness 核心。它學到的「原本哪裡錯」要寫成 correction,讓 Planner/Builder 下次避開 —— 這是 agent 之間透過記憶底層的間接學習。
  4. memory 一律「草稿 → 審核 → 版本化」。比照我寫 memory 前會查重、會刪錯的紀律,也比照你們 V54 衍生草案。自動寫記憶不設閘 = feedback 漂移。
  5. 短期 context ≠ 長期 memory,別混。working context(當前 build 的 observation)用完即棄;只有被蒸餾/被 feedback 認證的才升級成 durable memory。這是我 compaction 與 memory 的分界。

10Builder 的「文件備忘」機制

Builder 不寫散落 episodic,而是把每則學習貼成錨定 block_id(+選配 param)的備忘 —— 就是文件旁的便利貼,審核佇列,不是文件本身。

為什麼更好

錨定備忘直接掛在該 block doc 上,DOC_GAP 變「數數」而非模糊關聯
單一來源備忘永遠是候選,經 review 才畢業進 block_docs — doc 不被污染
可見健康度每份 doc 顯示「N 則待處理備忘」→ doc 自報弱點
自癒閉環備忘畢業 → Builder 讀到更好的 doc → 該 block 備忘變少

備忘要帶「結果脈絡」

帶 verdict(Planner 判對/錯、verifier verdict),Supervisor 才分得出兩種 doc 問題:

成功但卡了一下→ doc 要補強 / 講清楚(clarify)
失敗→ doc 有錯要修(fix)

實作:block_doc_memos 表(block_id/param/memo/verdict_context/from_build/status),BlockDocsDrawer 順便秀待處理備忘。跨 block 的序列配方仍走 episodic → procedure。

11.1 Supervisor 怎麼「修正」三個 Agent（based on log）

12B — 4 Agent 的記錄行為：情境 · Timing · 如何 Cue

核心原則:「何時記錄」由 graph 的 deterministic 事件決定,不靠 LLM 自由意志(否則會時記時不記)。 LLM 只負責「寫什麼」—— 填一個結構化 memo 模板;class 由決策樹選。這守住 flow-in-graph,agent 不會忘記記、也不會亂記。

correction 特化：{ 情境, 原本錯在哪, 正確做法, why, tag: plan|execute }

13C — Build Trace 要什麼（現行不足，尤其缺 User Feedback）

現行 BuildTracer 是 LLM-call log(每 call 的 node/prompt/token/verdict)—— debug LLM 很好,餵 Supervisor 學習卻缺三塊致命的。

你點名的關鍵場景:Planner & Builder 以為做好了(ok=true、verifier 過、Planner 判對),但 user 說「不是我要的」→ Supervisor 檢討。
要抓到它,trace 必須同時記兩個相反訊號並連結:系統自認 OK ＋ user 交付後否決 → 派生出 divergence。 當「所有內部訊號都說 OK、user 卻不買單」= 不是執行錯,是意圖沒抓對(plan 抓錯需求 / 漏呈現偏好 / 領域假設錯)—— 這是單看 verifier 永遠看不到的最高價值訊號。 
Trace 要從「LLM-call log」升級成「Episode 記錄」：
Episode {
  ids:        build_id · user_id · pipeline_id · scope(skill/tool/recipe)
  intent:     原始 user 需求
  plan:       proposed → accepted        (+ diff = plan_feedback)
  build:      [per-phase: blocks, params, verifier_verdicts,
               reject→fix 事件, inspect_mismatches]     ← 餵 doc 備忘
  self_assessment: ok · verifier_passed · planner_judge  ← 系統自認
  delivery:   result summary (charts / tables)
  user_feedback: [{ stage: plan|delivery|repair,
                    text, sentiment: accept|edit|reject, target }]  ← 關鍵新增
  repair:     [feedback → root_cause → fix]
  divergence: self_assessment=ok  BUT  user=reject       ← 派生 · 金礦
}
現行 llm_calls trace = Episode 底下一層(debug 用);Supervisor 主吃 Episode。
