Multi-Agent 架構 — 三平面模型

建置平面(Planner/Builder/Repair) · 監督平面(Supervisor) · 監控平面(機台/recipe/APC/站點) · 共用知識底層

1核心觀點:這是三個「時鐘」不同的平面

最容易犯的錯是把它們當成一套系統。它們的觸發、生命週期、狀態完全不同 — 分開設計才乾淨。

同步 · REQUEST-SCOPED

建置平面

Planner → Builder → Repair。協作把一條 pipeline 建出來。graph 當 supervisor、共用 canvas/phase 狀態。就是我們剛 spec 的角色拆分。

週期性 · 離線

監督 / 學習平面

Supervisor。不在熱路徑上;定期讀 traces + 歷程,對各 agent 提改進、甚至指出「這是 doc 不齊全」。輸出是寫回知識,不是改 flow。

排程 / 事件 · RUNTIME

監控平面

機台 / recipe / APC / 站點偏移 monitor。本質是 skills_v2 patrol(排程 skill + verdict + alarm) — 不是新框架,是既有 auto-patrol 基建。

2關係圖

實線 = request 熱路徑資料流;虛線 = 離線 / 週期性的觀測與回饋。知識底層是三平面共用的 single source。

3每個 Agent 的職責

Planner建置

專注擬定 plan:把常用 block、user 偏好整理好,產出更準的 phase 拆解,讓後面執行更清楚。

in: 意圖 + 偏好/kn識　out: phases
Builder建置

follow plan 建置。建的過程逐步熟悉積木的關係、特性、與參數怎麼調配(高頻、吃大部分 token,須配便宜+cache 友善模型)。

in: phases + canvas + kn識　out: canvas patch
Repair建置

前面建好後,幫忙再確認結果,並根據 user 的 feedback 給予修正。

in: 結果 + user feedback　out: 修正 patch
Supervisor監督·離線

定期 trace 各 agent 過去的表現與歷程,給各 agent 建議、幫忙修正,甚至指出「這是 document 不齊全」的根因。輸出是寫回知識(草案,人工 review)。

in: build traces + monitor 結果　out: kn識/doc/prompt 改進 draft
機台 / Recipe / APC / 站點 Monitor監控·runtime

監控機台狀態、recipe 表現、APC 參數 drift、站點是否偏移。每個 = 一條 skills_v2 patrol(排程/事件觸發 → verdict → alarm),不是新框架。

in: live 資料　out: verdict → alarm / 事件
Monitor-Correlator（選配）未來

真正值得做「agent」的是跨訊號關聯:把機台/recipe/APC/站點的告警融合成一次 RCA(單一 patrol 做不到)。這才是 monitor 平面的 agent 加值點。

in: N 個 monitor 訊號　out: 融合 RCA

4我的幾個關鍵想法

  1. 三平面別混在一起做。建置=同步/graph;監督=週期/離線;監控=排程/事件。它們的時鐘不同,混起來會又難測又難維護 — 這正是你要拆的初衷。
  2. Supervisor 的輸出要是「寫回知識」,不是「改 flow / 改 prompt」。它看到 Builder 老猜錯某 block 參數 → 判定 doc 不齊 → 產 block_docs / agent_knowledge 改進草案。這剛好落實你們原則 0(doc 是唯一來源)、又守住「flow 在 graph」。
  3. Supervisor 必須離熱路徑。建置要快、要 deterministic;Supervisor 是離線優化器,改的是 build agent「讀的知識」,不是「當下的流程」。
  4. 監控四兄弟 = skills_v2 patrol,不是新造框架。機台/recipe/APC/站點 monitor 就是四條排程 skill(verdict → alarm),既有基建(SkillRunner / block_step_check / alarms / event fan-out)直接用。
  5. 有個漂亮的閉環:建置平面產出監控 patrol → 監控產生訊號 → Supervisor 同時吃 build traces + monitor 結果 → 回頭改進建置。整個 lifecycle 收斂。
  6. 回饋要「有閘、有版本、可稽核」(小心不穩定)。Supervisor 自動改 agent 會有 feedback 漂移風險。走法比照 V54 衍生草案 + 角色稽核:proposal → 人工/gated review → versioned kn識,不做自主 prompt 突變。
一句話收斂:把「一個 agent」變成「三個平面 + 共用知識底層」。建置平面是 3 個 role agent(Planner/Builder/Repair,graph 編排);Supervisor 是離線學習迴路(寫回知識);監控平面是既有 skills_v2 patrol 的具體化。四者透過同一份知識底層鬆耦合。

5幾個要你拍板的開放問題

Q1 · Supervisor 的回饋要多自動?　全人工 review 草案 / 半自動(高信心自動、低信心排隊) / 只發「doc 不齊」報告不自動寫。
Q2 · 監控平面現在要不要動?　建議先只落地「建置三平面 + Supervisor 骨架」,監控維持現有 patrol,等建置穩了再做 Monitor-Correlator。
Q3 · Supervisor 的觸發源　只吃 build traces,還是一開始就併 monitor 結果?(併了才有跨平面學習,但範圍大。)
Q4 · 落地順序　先抽 Agent 契約 + 拆 Planner/Builder/Repair(純重構、SLASH-17 零回歸),Supervisor 當第二階段?
