打 Anthropic 模型：直連 vs OpenRouter vs AWS Bedrock

回答三個問題：通路之間差在哪、thinking 與 prompt cache 各通路怎麼開、 以及在我們平台上有哪些現成開關與地雷。含本平台 7 月實測數據佐證。

02Thinking 怎麼開 — 兩代 API 不一樣

這是本週實驗踩過的雷：Claude 5 家族與 Claude 4 家族的 thinking API 完全不同，用錯直接 400。

Claude 5 家族（sonnet-5 / fable-5）Claude 4 家族（haiku-4.5 / opus-4.8）
機制adaptive thinking，常開、關不掉；用 effort 控深度extended thinking，預設關；用 token 預算開
參數thinking:{type:"adaptive"}
output_config:{effort:"low|medium|high|xhigh"}		 thinking:{type:"enabled",budget_tokens:N}
限制不支援 temperature / top_p / 固定 budget / assistant prefill開 thinking 時 temperature 必須 1；max_tokens 必須 > budget
tool-use 迴圈回放 assistant 內容時 thinking block 必須原封含 signature 傳回，否則 400（我們的 client 已處理）
修正一個先前的說法：Sonnet 5 的 thinking 其實常開 — 7/6「快答版」跑的是預設 effort 檔， 7/7 顯式設 effort=high 才拿到 17/17。所以正確結論是「effort 檔位決定成敗」而非「開/關 thinking」。 另外還有 xhigh 檔沒測過，是後續可做的實驗。

03Prompt cache 怎麼開 — 要點與地雷

要點內容
斷點機制在 system / tools / messages 的 content block 加 cache_control:{type:"ephemeral"}，最多 4 個顯式斷點
最小可快取前綴約 1024–4096 tokens（依模型）— 太短的 prompt 不會進 cache
TTL 與價格5 分鐘（寫 1.25×、讀 0.1×）或 1 小時（寫 2×）— agent 迴圈這種連續呼叫是 cache 的最佳場景
驗證方式看回應 usage 的 cache_read_input_tokens 是否 > 0，不要假設有開就有效（OR 的教訓）
OpenRouter 特有官方建議 top-level cache_control 自動模式；但 OpenAI-wire 相容層是已知雷區，用前先量測
Bedrock 特有Converse API 用 cachePoint block；逐 region × model 查支援矩陣
我們的實測基準：直連 + agent 迴圈場景，Haiku 快答 cache 99.7%（fresh input 只 14.6k tokens、cache read 5.24M）， 17 案全量 $1.02。同場景走 OR 時 cache 0%，Sonnet 成本假性膨脹 ~5 倍。

04本平台現成開關（.env，已上線）

# python_ai_sidecar/.env — 單一切換點
LLM_PROVIDER=anthropic            # anthropic 直連 | ollama(=OpenAI相容, 打 OR/GLM)
LLM_MODEL=claude-haiku-4-5-20251001

# Claude 5 家族用（sonnet-5）：adaptive + effort
ANTHROPIC_THINKING_EFFORT=high    # low|medium|high（xhigh 未接，可加）；空 = 模型預設檔

# Claude 4 家族用（haiku-4.5 / opus-4.8）：budget 式
ANTHROPIC_THINKING_BUDGET=6000    # 0 = 關；>0 自動把 max_tokens 抬過 budget

# 只在完全沒開 thinking 時生效（API 規定）
ANTHROPIC_TEMPERATURE=            # 注意：sonnet-5 完全不支援 temperature，留空
提醒說明
兩個 thinking 開關別同時設EFFORT 優先；模型家族用錯參數會 400（錯誤訊息會告訴你該用哪個）
sonnet-5 + TEMPERATURE會 400 — 該模型不收 temperature，保持留空
cache 驗證習慣換通路/模型後第一件事：跑一案、查 episode cost_json 的 cache_read 是否 > 0
成本歸因thinking tokens 計入 output — Agent Activity 的 LLM 呼叫數/耗時已可觀測，比對 effort 檔位成本用它
若要接 BedrockSDK 有 AnthropicBedrock client，llm_client 加一個 provider 分支 + IAM 認證即可，工作量約半天；建議等「進廠/合規」需求明確再做

05建議

情境建議通路理由
現在（開發/驗證）Anthropic 直連cache 實測最好、thinking 第一手、Day 0 新模型；已驗證 $1–2.3/全量 17 案
打非 Anthropic 模型（GLM 備援）OpenRouter它的價值在聚合多模型，不在打 Anthropic
進廠 / 合規 / 私網AWS BedrockVPC/PrivateLink + AWS 帳務整合；功能與官方同步，遷移成本低（半天接一個 provider 分支）
永遠別做OR 打 Anthropic 模型當 defaultcache 靜默失效 = 成本 5 倍，且不易察覺（HTTP 全 200）
