D成本現實：cache 補測後的公平比較

bakeoff 當日只有 GLM 的 cache 生效（KIMI@Fireworks 0%、Anthropic 走 OpenRouter 的 OpenAI-compat 路徑會抹掉 cache 斷點 → 0%）。這對 Anthropic 兩個成本不公平。
2026-07-02 補測（direct Anthropic API，cache 全開）： Opus cache 51.8% → $52.10（原 $68.6，−24%）；Sonnet cache 63.2% → $11.58（原 $18.7，−38%）。 cache 確實生效,但成本沒砍半 —— 17 個獨立 build 各自 cold-start 快取、要付 cache-write（Opus 寫入 $18.75/M），跨 build 又超過 5 分 TTL 無法共用；production 連續 session 會攤更好，這是 benchmark 下限。 結論不變:即使給 Anthropic 最有利的 direct+cache，Opus 仍 ~20× GLM、Sonnet ~5× GLM。

E結論與定位

  • GLM-5.2 — prod 預設（不動）。嚴格品質與 Opus 僅差 1 案（xbar_r 也做對），成本最低、cache 唯一有效、速度屬最快 tier。CP 值完勝。
  • Opus 4.8 — 品質天花板，僅作「難題升級層」。嚴格 16/17 最高、最果斷（164 round）。direct+cache 實測 $52（~20× GLM）—— 只在 fleet / composite / 特定管制圖等難案人工升級時叫用，且務必走 direct Anthropic 才有 cache。
  • Sonnet 5 — 無角色（被 GLM 支配）。嚴格 15/17（xbar_r 還退化），direct+cache 成本仍 ~5× GLM。比 GLM 貴又不比它準，沒有採用理由。
  • KIMI K2.5 — 已淘汰。最慢（73 min）、cache 0%、品質不優於 GLM；2026-06-25 已被 GLM 取代。
  • 共同待辦：weco_rules（apc-drift）四模型皆弱 → 修 knowledge / block doc，換模型無效。
