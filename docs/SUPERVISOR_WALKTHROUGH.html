三個 Agent 的筆記，與 Supervisor 的治理

寫給 product owner 的版本：用一個 request 的生命週期，說明 Planner / Builder / Repair 各自留下什麼筆記（W1 / W2 / W3）、什麼時候生效、Supervisor 在哪一刻介入 — 以及它永遠不做什麼。

對應介面：/supervisor 簽核工作台 · /agent-knowledge 知識工房 · /me 我的偏好

先回答最常見的誤解：Planner 記下你的修正後，要等 Supervisor 審核嗎？— 不用。

Supervisor 不是流程中的守門員，它是事後的策展人。三種筆記的治理等級由「訊號可信度」決定：

筆記誰寫的訊號來源生效方式Supervisor 的角色
W1 偏好Planneruser 親手修改 plan立即生效事後整理：去重、清理、抓矛盾
W2 便利貼Builderagent 踩坑的自述進待審佇列，人審才生效彙整成提案、起草修訂稿
W3 修正Repairagent 修復後的推論draft，不參與檢索主動提案轉正，人簽核

可信度遞減（親手 > 自述 > 推論），生效門檻就遞增（即時 > PE 審 > 提案＋簽核）。

一Planner 與 W1 — 偏好（preference）

「這個 user 喜歡怎樣」。個人 scope — 只影響你自己的 build，存在你名下，/me 可自助管理。

類別定義W1 是什麼

Planner 在 plan 確認關卡觀察你的編輯行為。你改了它提的 plan，就代表它對你的理解有落差 — 這是全系統可信度最高的訊號：不是 agent 猜的，是你親手改的。所以 W1 寫入即 active，不設審核。

情境「SPC 圖要附 UCL/LCL」

你
說「幫我看 EQP-01 的 SPC 趨勢」。
P
PLANNER提出 plan：p1 撈資料 → p2 聚焦 xbar → p3 畫趨勢圖。暫停等你確認。
你
在確認卡上把 p3 改成「趨勢圖，要附 UCL/LCL 管制線」，按確認。
✓
兩件事同時發生、互不等待：(a) build 照你改過的 plan 繼續跑；(b) Planner 寫下 W1。
◆
W1 寫入「這個 user 的 SPC 圖要帶 UCL/LCL」 — 寫完即生效，Console 活動流可看到這筆 memory_write。
✓
下次你再要 SPC 圖：Planner 召回這筆 W1，第一次提的 plan 就自帶管制線。全程沒有任何人審核過這筆記憶。

Supervisor 介入不在當下，在每週體檢

Supervisor 定期巡視所有 W1，只做三種提案（送進工作台，人簽核才落地）：

PRUNE
這筆偏好 90 天沒被召回 → 提案清掉（軟刪 30 天可復原）
MERGE
你前後留了兩筆講同一件事的偏好 → 提案合併、保留較完整的那筆
矛盾偵測
你的偏好與新知識條目衝突（同時召回會互打）→ 提案讓人裁決留哪個

你自己隨時可以在 /me 我的偏好 停用或刪除 — 不必經過任何人。

二Builder 與 W2 — 文件便利貼（doc memo）

「block 文件沒講清楚的坑」。不是知識條目，是貼在某個 block 文件上的便利貼 — 因為 block docs 是 LLM 的唯一文件來源，agent 不准直接改它。

類別定義W2 是什麼

Builder 在建構中踩了坑又爬出來（被 Verifier 拒絕 → 修正 → 通過），這段「拒→修→過」的經驗如果源頭是文件缺陷，Builder 就寫張便利貼。它是 agent 對自己失敗的自述（△ 虛線級訊號），而且對象是全域共用的文件 — 錯了會污染所有人，所以必須人審。

情境「limit 到底是字串還是數字」

B
BUILDER對 block_unnest 設 limit="25"（字串）。
V
VERIFIER結構檢查（零 LLM）拒絕：PARAM_TYPE_WRONG — limit 應為 integer。
✓
BUILDER下一輪改成 limit=25，通過。
◆
W2 寫入「block_unnest 的 limit 型別易誤，文件應標明 integer」 — 進 review queue。不生效、不改文件，只是躺在佇列裡。

Supervisor 介入彙整 + 起草，送 PE 簽核

S
SUPERVISOR發現同一個 block 累積了 4 張同主題便利貼 — 不是偶發，是文件真的有缺口。彙整成一個 DOC_REVISE 提案，並把修訂後的文件段落草稿起好，附證據鏈：▣ episodes ×4▣ 拒→過 diff
PE
在知識工房收件匣看到這張卡（三段式：提案／為什麼／依據），核准（可先改寫草稿）或駁回。
✓
核准 → block docs 更新。之後所有 user 的 build 讀到修訂後的文件，這個坑不會再有人踩。

三Repair 與 W3 — 修正（correction）

「這樣修才對」的推論。比 W2 更危險 — 是 agent 從一次修復推廣出來的通則，可能只是巧合，所以門檻最高。

類別定義W3 是什麼

Builder 連續卡關時 Repair 介入診斷、修復。Repair 把修復心得寫成 W3 — 但它存成 draft：檢索完全撈不到它，等於這筆知識還不存在。原因：單一案例的修復推論，未經其他證據支持前不能污染知識庫。

情境「spec_limits 要用巢狀物件」

V
VERIFIER對 p3 的 block_spc_panel 連拒兩次（缺 spec_limits）。Builder 卡住，升級 Repair。
R
REPAIR三路對齊診斷（原始需求 × canvas × 拒因），重建節點、把 spec_limits 改成巢狀物件，通過。
◆
W3 寫入「spc_panel 的 spec_limits 需要巢狀物件格式」 — draft，不參與任何檢索。

Supervisor 介入找旁證，提 PROMOTE，人簽核

S
SUPERVISOR不會放著 draft 爛掉。它比對其他訊號：本週 3 件「agent 自評 PASS 但 user 打不準確」全落在同一情境，且該步驟知識召回 0 筆（知識缺口實錘）。於是提 PROMOTE 提案：「把這筆 draft 轉正為 domain 知識」，附證據鏈：▣ 召回 0 筆 ×3△ user 評「不準確」×3◆ 對應 W3 draft
PE
在收件匣核准（How-to-apply 草稿可先改寫）。
✓
30 秒 embed 完成 → 下次 build 開始被召回。從「一次修復的推論」變成「全域生效的知識」，中間隔著 Supervisor 的旁證 + 人的簽核。

總所以 Supervisor 到底是什麼

把三章疊起來看：agents 在戰場上留下三種筆記，可信度遞減、生效門檻遞增。Supervisor 是定期巡視筆記堆的策展人 — 把散落的訊號彙整成「有證據的提案」，但落地的手永遠是人的。

2026-07 起它的職掌從知識面擴到執行面（監督三面）：

監督面看什麼產出簽核者
S1 執行健康build 成功率、rounds、拒因分布、退步偵測診斷 + 調參提案 / issue 型提案IT_ADMIN / 開發者
S2 知識品質召回命中、W1–W3 佇列、零召回事件PRUNE / PROMOTE / MERGE / DOC_REVISEPE
S3 成本資源花費、cache、provider 品質（如空回應率）異常警報 + 設定調整提案IT_ADMIN

四條硬規則：無落地權（一切經人簽核，危險動作僅限 authed UI）、提案必附證據鏈（▣ 系統事實 / △ 自述 / ◆ 記憶，可點回 trace）、code 不在職掌（只指出、不修）、提案會過期（新證據出現自動 supersede）。

介面對照：提案簽核在 /supervisor（IT_ADMIN）；W2/W3/教它草稿的審核在 /agent-knowledge 知識工房收件匣（PE）；個人 W1 在 /me 我的偏好；每次 build 用了哪些記憶，看 Agent Console 的「記憶效應」區。
