Skill Library POC — 功能規格

產品功能說明（What it does） ·  部署細節見 POC_DEPLOYMENT_SPEC.html  ·  接手計畫見 POC_HANDOVER_PLAN.html  ·  2026-06-24

產品定位：給半導體運維工程師的「AI 分析技能庫」。工程師用自然語言提出運維問題 （例：「列出最近 7 天 OOC 最多的 3 台機台」），平台的 LLM Agent 自動把它轉成一條資料分析 pipeline、 對資料源執行、回傳圖表 / 表格 / 判定；好用的分析可發布成 Skill 給整個團隊重複使用。 目標是把「會寫 SQL / 會接資料的少數人」的能力，變成每個工程師都能用一句話取得。

1核心概念（領域語彙）

Skill（技能）

一條可重複使用、已發布的分析 pipeline，回答某一類運維問題。使用者選 Skill + 填參數即可得到結果。

Block（積木）

最小操作單元（取資料 / 轉換 / 畫圖 / 判定）。內建 56 個，每個自帶完整文件（用途、參數、範例）。

Pipeline（管線）

多個 block 串成的有向圖 —— 一個 Skill 的實作。由 Agent 自動產生，或手動拼。

System MCP（資料源）

admin 設定的外部 HTTP 資料來源。Skill 透過它取得真實資料；機密以 ${ENV} 引用、不存 DB。

Agent（代理）

把自然語言需求轉成 pipeline 的 LLM。會規劃、選 block、自我檢查與修正。

Knowledge（知識層）

規劃提示 + 領域模型 + block 文件，讓 Agent 接地（知道資料長相、該用哪個 block）。

2主要功能

L1

Library — 技能庫與目錄

瀏覽 / 搜尋已發布的 Skill 與 Block 目錄。每個 block 有「文件抽屜」秀用途 / 參數 / 範例。

使用者做什麼
  • 找現成 Skill 直接用
  • 查某個 block 怎麼用、輸出什麼
得到什麼
  • 可立即套用的分析
  • 清楚的 block 能力說明（人與 Agent 同一份）
L2

Authoring — 兩種建立方式

把需求變成一個 Skill 草稿。

A · Glass Box（自然語言）
  • 用一句話描述需求，Agent 自動規劃→選 block→建構→自我修正
  • 過程透明：即時看到它的 plan 與每一步在做什麼
  • 適合不熟 block 細節的工程師
B · Manual Builder（手動）
  • 在畫布上自己拖、接、設定 block
  • 完全掌控；適合進階使用者微調
  • 有 Block Docs 抽屜輔助設定
兩種方式產出的都是草稿 Skill；確認 + 試跑後可發布給團隊。
L3

Try-Run — 試跑與結果

把草稿 Skill 對真實資料跑一遍，當場看結果，確認對了再發布。

輸出型態
  • 圖表（chart）
  • 表格（table）
  • 單一數值 / 判定（scalar / verdict 過或不過）
支援的圖
line / barbox plotpareto X-bar/R 管制圖 + WECOEWMA-CUSUM 漂移Q-Q 常態
輔助功能做什麼
Block Docs自帶文件的 block 目錄。同一份描述同時給 Agent（選對 block）與 UI（給人看）—— 單一真相來源，不會人/機看到不同版本。
Build Trace每次建構都記錄 Agent 的 plan + 每步推理 + 驗證判定，可在後台回看「它為什麼這樣建」，用於除錯與信任。
System MCPadmin 連接外部資料源（endpoint / method / headers / input schema）。Skill 即可從中取數；機密用 ${ENV} 引用。

4Agent 怎麼把「一句話」變成「跑得出結果的 pipeline」

理解意圖→ （模糊則反問）→ 規劃 phases→ 確認→ 逐步建構＋自我檢查→ 執行→ 呈現結果
環節Agent 做的事（功能語意）
理解意圖判斷使用者要「查資料 / 比較 / 排名 / 概念問答」哪一類；資訊不足會主動反問澄清，不亂猜。
規劃把需求拆成「想達到的階段性結果」（取資料 → 轉換 → 圖表/判定），先講意圖、不先綁工具。
建構 + 自我檢查逐 phase 挑 block、設參數；每步做結構檢查（例：沒接好的懸空節點會當下抓出來修），確保產出跑得動。
穩定性外部模型偶發抖動會自動重試；參數型別偶發寫錯會自動修正 —— 一次小失誤不會讓整個建構失敗。
呈現依需求輸出圖 / 表 / 判定，並把建構軌跡留存供回看。

兩個入口：Chat 面板（運維對話問答）與 Builder Glass Box（建構技能）。流程控制走確定性的 graph node，不依賴 prompt 自律 —— 行為可測、可回歸。

5角色與權限

角色能做什麼
IT_ADMIN全部 + 使用者/角色管理、System MCP 設定
PE（製程工程師）編寫 / 建構 / 發布 Skill、設定、跑分析
ON_DUTY（值班）使用已發布的 Skill、讀資料；不可建構新 pipeline 或寫入全域記憶
已知產品待決策：值班帳號問「臨時數據問題」時，chat 目前會導去「建 pipeline」確認流程，但值班無建構權 → 拿不到答案。 是否讓值班改走「直接答 / 跑已發布 skill」是待 stakeholder 拍板的 UX 決策。

6品質與成熟度（可量測）

平台準確度有客觀基準、可回歸，不是憑感覺：

面向現況
Builder 準確度SLASH-17 回歸套件：17 條代表性指令對「人工標準答案」評分（建出的 pipeline 對不對）。目前接近 17/17。
Chat 準確度chat-mode eval harness：驗「確認後真的建出 pipeline + 跑出結果」。
穩定性provider 抖動重試、結構自我檢查、參數自動修正 —— 已用多輪重跑驗證收斂。
除錯能力Build Trace + trace replay：可重現、可解釋每個失敗在哪一步。
