Architecture

5 services across 4 ports. Each owns a narrow slice; cross-service calls are explicit HTTP with auth tokens.

Data flow examples

1. User chats "查 EQP-08 OOC"

browser → aiops-app /api/chat → sidecar /internal/agent/chat (NextAuth session) (X-Service-Token) │ ▼ chat orchestrator_v2 (LangGraph) │ ▼ intent_classifier → tool_call bucket │ ▼ LLM picks skill → call java /api/v1/skill-documents/{slug}/run │ (forwarded JWT from session) ▼ java SkillRunnerService → POST sidecar /internal/pipeline/execute │ ▼ block executors run in-process (process_history → ontology_simulator via HTTP for raw data)

2. Build agent constructs a pipeline

browser → aiops-app /admin/pipeline-builder → aiops-app /api/agent/build → sidecar /internal/agent/build │ ▼ classify_advisor_intent → BUILD bucket │ ▼ stream_graph_build (LangGraph state machine) ┌─────────┴─────────────┐ │ │ ▼ ▼ goal_plan_node agentic_phase_loop ⇄ phase_verifier (LLM: 3-5 phases) (LLM-driven ReAct, sub-phase machine) │ ▼ final_pipeline → save via Java

3. Automated skill fires by schedule

java-scheduler @Scheduled (every 60s) → SkillScheduleService.tick() │ ▼ scan skill_documents WHERE status=stable AND trigger_config.type=schedule AND due │ ▼ SkillApiClient.dispatchSkill(slug, "system_schedule") │ (X-Internal-Token to java-backend) ▼ java-backend /internal/skills/by-slug/{slug}/run-system │ ▼ SkillRunnerService.run() → triggered_by="system_schedule" │ ▼ executes pipeline → block_alert writes to alarms table
