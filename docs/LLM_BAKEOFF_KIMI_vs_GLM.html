A我們 Agent 對 LLM 的特性需求

先講清楚 builder 是什麼樣的 agent —— 一個 graph-driven、逐 round 動作的 ReAct agent，它對 LLM 的要求和一般 chat 不同。

#特性需求為什麼 critical
1逐 round function calling每 round 只發 1 個 tool call（add_node / set_param / connect…）；tool-call 不穩整條建構就崩
2結構化 JSON 輸出goal_plan 出 3-7 phase JSON、task_contract 出結構欄位；parse 失敗就 retry / fail
3長多步推理（≤32 round/phase）每 round 看一個 13 段的大 observation（含整個 block catalog），要跨 round 維持連貫
4中英混雜prompt + 機台指令雙語（「中位數 / median」「偵測 / 找出」同義變形）
5composite / fleet block 接線list_objects→mcp_foreach（全廠 fan-out）、spc_panel（複合）— 最難的推理
6prompt-cache 友善（成本）那個大 observation prefix 每 round 重複；cache 命中率 = 成本主槓桿
7果斷、不過度建構該用 minimal pipeline，別亂加 node（verbose catalog 會誘發 over-build）
8自我修正回應 verifier reject、改參數 / 重接線，不要繞圈

D結論

  • 嚴格標準（golden 全中）：兩者通過數接近（~14/17），KIMI 穩定、GLM 12-15 有 run-to-run 變異。
  • 寬鬆標準（功能可用）：GLM 17/17 略勝 KIMI 15/17 — GLM 修掉 KIMI 兩個實質錯，而 GLM 的失分多為 golden 過嚴或系統共通弱點（sort / fleet，KIMI 也中）。
  • 效率：GLM 少 29% 呼叫、跑完全套快 61%（28 min vs 73 min）；最極端 apc-trend 為 5 call/44s vs 40 call/533s。
  • 成本：KIMI@Fireworks 實測 cache=0，GLM 40-58% → GLM 成本不輸還更穩定。
  • 決策：已切換 prod 至 GLM-5.2 @ Fireworks @ medium。唯一要管理的是變異，將以 hardening（phase 拆原子 / fleet hint / spc_panel fallback）壓低 — 這些對 KIMI 同樣有益。
