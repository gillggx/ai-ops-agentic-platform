各 Agent 詳細運作流程

Planner / Builder / Verifier / Repair / Supervisor / Monitor — 觸發時機 · 輸入輸出 · 逐步流程 · 記憶讀寫 · 預算與護欄 · 觀測事件。以 2026-07-04 上線 code 為準。

0全景：一次 build 的生命週期

三個 build-plane agent（Planner / Builder / Repair）住在同一張 LangGraph 裡，由 graph 決定流程、agent 只做窄任務（hard rule：flow 不進 prompt）。Verifier 是 deterministic node，永遠不是 agent。Supervisor / Monitor 在 build 之外離線運作。

   │ phases: P1..PN（意圖，不含 block）
   ▼

（離線，不在上圖內）
SUPERVISOR：讀記憶+備忘 → 蒸餾提案 → 人工核准 → commit
MONITOR  ：掃 episode 指標 → 改善請求 → 人工核准 → 產 Planner 指令
設計不變式：(1) 流程由 graph node 決定，LLM 不決定下一步（CLAUDE.md hard rule）。(2) Verifier 永不 agent 化 — 結構正確性必須可單測、可重現。(3) 任何「寫回記憶/文件」的自動行為都是 propose-only，人核准才生效（2026-07-03 污染事件教訓）。

1PLANNER — 需求擁有者 + 語意裁判

Planner python_ai_sidecar/agent_builder/agents/planner.py · 委派節點 goal_plan / (Phase 2+) 裁判
Charter
「pipeline architect。把 user 需求拆成 goal-oriented phases（意圖，不挑 block）；Phase 2+ 起並擔任語意裁判（APPROVE / REVISE / REPLAN）。」
觸發時機
每次 build 的第一個節點（goal_plan）；REPLAN 時帶 v30_replan_hint 重入。
Model
繼承 session default（現為 GLM-5.2 @ Fireworks）。
工具
無 — 只輸出 JSON（phases），不碰 canvas。
預算
replan_count=2（react/revise/repair 均為 0 — 不是它花的）。

逐步流程

#步驟細節
1讀 state view只讀 5 個欄位：instruction / base_pipeline（增量建置時的既有 canvas）/ skill_step_mode / v30_replan_hint / user_id。
2知識注入（讀記憶）build_knowledge_hint(source="goal_plan", agent="planner", round=0)：always_on 高優先知識直推 + RAG（pgvector, layer=plan）召回 preference / presentation / correction / domain。召回清單以 memory_recall 事件記錄（/agent-activity 可見每輪引用了哪幾條）。
3產 plan1 次 LLM call 輸出 P1..PN，每 phase 只有 3 欄：id / expected（資料型態）/ 一句 text — block-agnostic（挑 block 是 Builder 的事）。provider error 有 bounded retry（2 次）。
4Confirm gate（graph node，非 LLM）plan 呈現給 user；user 可逐 phase 改文字。W1 fast-path：每筆修改經 classify_edit（deterministic regex 樹：presentation > preference > correction）分類後寫入 agent_knowledge（written_by=planner, active=true, 上限 3 筆/build）— 「user 明改」是最高可信訊號。
5裁判職（Phase 2+ 位置）phase 完成後判 APPROVE / REVISE / REPLAN；REPLAN 消耗 replan 預算重入 goal_plan。

記憶 × 觀測

讀寫觀測事件
domain / preference / presentation / correction(tag=plan) / procedure — always_on + RAG W1：preference / presentation / correction（fast, active） plan_proposed · plan_user_edited · plan_confirmed · memory_recall · llm_usage（per-agent 成本歸因）

2BUILDER — 執行者（ReAct loop）

Builder agents/builder.py · 委派節點 agentic_phase_loop
Charter
「follow Planner 的 phase 逐一執行：選 block、接線、設參（param key 100% 依 param_schema）。每 round 一個 tool call；結構驗證由 deterministic verifier 把關。」
觸發時機
plan 確認後，每個 phase 進入 agentic_phase_loop。
工具（8）
inspect_node_outputinspect_block_docadd_nodeconnectset_paramremove_noderun_verifierphase_complete
預算
react_rounds=32/phase（生產值，界定單 phase 成本上限）。

單一 round 的解剖（× 最多 32）

round N：
 1. 組 prompt：charter + phase 目標 + canvas 現況 + 上輪 verifier 拒因(結構化)
    + 知識注入（always_on + RAG layer=execute；agent="builder", phase_id, round
      → memory_recall 事件）
    + RAG 查詢工具（query_blocks / query_columns / query_connectable_sources
      — 「目錄不是教科書」，catalog 每 block ≤1500 chars）
 2. LLM 輸出 恰好一個 tool call（多 call / 無 call 都由 loop 矯正）
 3. graph 執行該 canvas op（deterministic dispatch — 不是 LLM 自己執行）
 4. op 結果（含 preview / error）進下一 round 的 observation
 5. Builder 認為 phase 完成 → phase_complete → 交 verifier

關鍵行為規則

規則為什麼
先 inspect_node_output 上游、再接線設參不 inspect 就會幻覺輸出欄位形狀（narrative 重建的常見 Layer-1 錯誤）。
param key 100% 依 inspect_block_doc 的 param_schemablock description 是唯一文件來源（CLAUDE.md 原則 1）。
每個 data node 必須有下游 consumerverifier 會拒 orphan leaf — 這條已由 Supervisor 蒸餾成 procedure 記憶。
不讀 preference（隔 Planner 一層）user 偏好在 plan 層就已折進 phase 文字。

記憶 × 觀測

讀寫觀測事件
domain（多在 block_docs）/ presentation / correction(tag=execute) / procedure W2：phase 曾被 verifier 拒 ≥1 次後通過 → block_doc_memos 寫一張「文件便利貼」（review queue，絕不直接改 block_docs；上限 5 筆/build） phase_started · block_picked · verifier_reject · param_reject_fix · phase_done · stuck_escalated · memory_recall · llm_usage

3VERIFIER — deterministic 守門員（不是 agent）

Phase Verifier graph node phase_verifier（phase-spanning）· 永不 agent 化

檢查層次

層檢查手段
結構orphan node / 斷線 port / param 型別（PARAM_TYPE_WRONG）/ 必填缺漏pure function，零 LLM
執行試跑到 phase terminal，rows>0、欄位存在、chart spec 成形sidecar in-process executor
語意結果是否符合 v30_task_contract（confirm 後由 extractor 萃取的驗收 contract）；量化 deficit（rows < 80% target → 暫停開 JudgeClarifyCard 問 user）contract 比對優先，fallback LLM-judge

裁決與去向

verdict:
  ADVANCED  → 記 phase_done，前進下一 phase（1-block phase 可自動合併多 phase）
  REJECTED  → 結構化拒因{block_id, reason, missing_for_phase}回 Builder 下一 round
              （這份拒因同時是 W2 doc memo 與 Supervisor DOC_GAP 分析的原料）
  round 用盡 → 交 REPAIR（phase_revise）
為什麼永不 agent 化：驗證是系統的正確性底線 — 必須可單測、可重現、不受 LLM 心情影響。出錯時 graph 知道是哪個檢查 fail；若驗證交給 LLM，只能猜「它又走偏了」。

4REPAIR — feedback 驅動的修正者

Repair agents/repair.py · 委派節點 phase_revise
Charter
「對齊三路輸入：原始需求、目前 pipeline 結果、user feedback。先診斷層級（build_level → 直接 patch canvas；plan_level → 產修正 plan 交回 Planner），再最小範圍修正。」
觸發時機
(a) phase round 用盡（stuck_escalated）；(b) Phase 2+ user feedback 開 repair ticket。
工具（6）
inspect_node_outputadd_nodeconnectset_paramremove_noderun_verifier（無 inspect_block_doc / phase_complete — 修正不重規劃）
預算
revise_attempts=1/phase · repair_iterations=3（ticket loop 上限）。

逐步流程

#步驟細節
1三路對齊state view：instruction（原始需求）+ canvas + last_verifier_reject + repair_ticket + phase_outcomes。
2根因診斷1 次 LLM call 輸出 {root_cause, alternative_strategy, missing_capabilities}；判定 build_level（param / 接線 / block 選錯）vs plan_level（phase 目標本身錯）。
3最小修正build_level → 直接 patch canvas（注意教訓：repair LLM 傾向「加 n1b 不移除壞 n1」— 需明確 remove_node）；plan_level → 產修正 plan 交回 Planner（REPLAN）。
4收場result=retry（修好，回 loop 續跑）或 result=handover（自我修復失敗，標記交棒續跑其餘 phase）。
5W3 寫記憶（僅 retry）修好才寫 correction 到 agent_knowledge（written_by=repair），且 active=false 草稿 — 等 Supervisor CORRECT 審核轉正。handover 絕不寫（污染實驗證實：live 失敗筆記在 execute 層召回會把同類 build 3/3 翻成 0/3）。

記憶 × 觀測

讀寫觀測事件
correction（避免重犯）/ domain / preference / presentation / procedure W3：correction 草稿（active=false，僅 retry 成功時） repair_triggered · repair_outcome{retry|handover} · llm_usage

5SUPERVISOR — 記憶蒸餾者（離線，propose-only）

Supervisor python_ai_sidecar/supervisor_curation/ · tools/supervisor_curate · Java SupervisorCurationService · /supervisor 頁
定位
把大量 episodic 原料（fast-path 記憶、doc 便利貼、episode 統計）蒸餾成乾淨、去重、分類的 durable memory — 它是 domain / procedure 兩類記憶唯一的產生者。
觸發時機
離線 CLI（tools/supervisor_curate）手動或 cron；絕不在 build hot path。
Model
Haiku 4.5 優先（cost posture 同 mcp_derivative）；無 Anthropic key 時回退部署 client（現為 GLM）。

Pipeline（一次 curation pass）

1. GET /internal/supervisor/curation-input（Java 彙整）
     draft corrections（W3 草稿）· live preferences / presentations（重複候選）
     · pending doc memos（W2 便利貼）
2. 1 次 LLM call：原則式 prompt（寧缺勿濫 / id 不可捏造 / per-type 精確 JSON schema）
3. deterministic validation gate：型別、必填欄位、每個 id 必須存在於輸入
     — 幻覺 id / 自創欄位一律丟棄（首次實跑擋下 11/11 自創格式，gate 有效）
4. POST /internal/supervisor/proposals → supervisor_actions（status=proposed）
     上限 12 提案/run · 同 (type,targets) 去重
5. 人工在 /supervisor 逐筆審核 → approve 才由 Java 在同一交易 commit

五種操作（approve 時的 commit 行為）

操作輸入commit 效果
MERGEkeep_id + remove_ids (+merged_body)保留者可更新內文；其餘 active=false。只合語意相同的 preference/presentation。
CORRECTtarget_id + new_body (+new_title, promote)把 W3 草稿改寫成乾淨教訓（含 Why / How to apply）；promote=true 才轉 active。
PRUNEtarget_ids過時 / 一次性 / 無再用價值 → active=false。
PROMOTEmemo_class(domain|procedure) + title + body + applies_to新增一筆 durable 記憶：written_by=supervisor、source=supervisor、active — 30 秒內 embedding backfill 後即可被 RAG 召回。只准 domain / procedure（fast-path 類不走此路）。
DOC_REVISEblock_id + memo_ids + revised_doc_draft相關 doc memos → promoted；修訂草案留在 action row — block_docs 本體不動（單一事實來源）。
Hard rule：propose-only。reject = 只蓋審核章，DB 一個 byte 都不動。Supervisor 永遠沒有直接寫記憶的路徑 — 這是 2026-07-03 污染事件（自動寫入的失敗筆記讓同類 build 3/3→0/3）換來的架構決定。

實跑紀錄（2026-07-04 首航）

13 筆提案入佇列（PROMOTE 4 / CORRECT 3 / PRUNE 3 / DOC_REVISE 3）；deterministic gate 另擋 4 筆幻覺格式
approve PROMOTE → 系統第一筆 domain 記憶（id 64）：從 2 筆獨立 correction 蒸餾出「limit 參數必須是 integer」領域事實
它也自己發現了 orphan-leaf 是跨 6+ block 的穩定 pattern，提案 PROMOTE 成 procedure — 與人工分析一致

6MONITOR — 自省觸發器（deterministic，無 LLM）

Monitor Java MonitorService · /supervisor 頁「改善請求」區 · V73 monitor_requests
定位
監控我們 agent 自己的健康（不是 fab 資料 — 那是 auto-patrol 的事）；把系統性劣化轉成「對 Planner 的改善請求」。與 Planner 同級的 requester，不是上級。
觸發時機
POST /api/v1/monitor/scan（UI「立即掃描」按鈕 / cron）。
LLM
零 — 三條指標全是 SQL 聚合 + 門檻，完全可重現。

三條指標

kind量測門檻訊號意義
DOC_GAP單一 block 的 pending doc memos 數≥ 3agent 反覆在同一 block 踩坑 → 文件缺口（per-block 一筆請求）
DIVERGENCE7 天內「自評 OK 但 user 否決」的 build 數≥ 2verifier / plan 品質漂移 — 系統最危險的盲區訊號
REPAIR_HANDOVER7 天內 repair 以 handover 收場次數≥ 3自我修復系統性失效（能力缺口 / doc 缺口 / plan 過度切分）

流程

scan（deterministic SQL）
  → 超標 → monitor_requests（status=open；(kind,subject) partial unique index 去重）
  → 人工在 /supervisor「改善請求」審核
      approve → 取得預備好的 Planner 指令（一鍵複製，拿去 Builder 下給 Planner）
      dismiss → 蓋章關閉

實跑紀錄（首掃）

DOC_GAP: block_spc_panel、block_weco_rules（各 3 筆 pending memo）
REPAIR_HANDOVER: 7 天 14 次（threshold 3）— 真實抓到 handover 偏高，佐證了 gate 期間 block_select bug 的存在

7.2 讀取管線（所有 agent 相同）

build_knowledge_hint(agent, phase_id, round)
  ├─ always_on 高優先列直推（不經 RAG）
  ├─ RAG：pgvector cosine（Cohere embed-multilingual-v3.0），layer 過濾
  │   （plan / execute / both — V58：錯層的列不進排序）
  └─ 召回清單 → memory_recall 事件{agent, phase_id, round, recalled[]}
      → /agent-activity Tab A 每輪 prompt 旁的紫色 chips

7.3 觀測（Episode = Supervisor / Monitor 的燃料）

層內容
agent_episodes每 build 一列：instruction / plan / status / 自評 / user feedback / divergence（自評 OK ∧ user 否決）/ cost_json（per-agent 歸因）/ trace_file 指標。
agent_steps行為事件流（14+ 類，上文各 agent 表列）；ContextVar 注入、fail-open、批次 flush（25）。
視覺化/agent-activity：Tab A Trace 明細（逐 phase/round prompt+output+引用記憶）/ Tab B 時間軸 / Tab C 記分板。

7.4 成本與旗標

項目設定
Build LLMGLM-5.2 @ OpenRouter（OLLAMA_MODEL，provider pin Fireworks — 快 8 倍）；per-agent 用量由 _observe_llm_usage hook 歸因。
Supervisor LLMHaiku 4.5 優先（離線、低頻）；SUPERVISOR_CURATION_LLM_MODEL 可覆寫。
Monitor零 LLM。
Feature flagsENABLE_AGENT_EPISODES / ENABLE_MEMORY_WRITES（env + restart；SSE endpoint 不吃 header 旗標）。兩者 OFF 時系統退回純 build 行為 — 全部旁路 fail-open。

7.5 學習閉環（把六個角色串起來）

build 失敗/被糾正
  → fast path（秒級）：W1/W2/W3 落 raw 記憶（planner 立即生效、repair 進草稿、builder 進便利貼）
  → distill path（天級）：Supervisor 蒸餾（合併/改寫/汰舊/升格/文件草案）→ 人核准
  → self-watch（週級）：Monitor 掃聚合指標 → 改善請求 → 人核准 → Planner 指令
下一次 build：RAG 召回更乾淨的記憶 → memory_recall 事件可回查「這輪引用了什麼」
