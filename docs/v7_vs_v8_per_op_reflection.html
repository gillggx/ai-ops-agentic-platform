Pipeline Builder Self-Correction：v7 (current) vs v8 (per-op reflection)

2026-05-13 — 詳細對照「整 plan 反思」與「per-op 反思」兩種架構，每一步逐項列出 Agent（graph node / deterministic state mutation）與 Model（LLM call）的工作。
AGENT 確定性 graph node，state mutation，無 LLM MODEL LLM call（Anthropic API） NEW v8 才有的步驟

1. TL;DR — 兩版設計的核心差異

v7 (current, shipped)

反思時機：整 plan 跑完 → finalize → inspect → reflect

反思 scope：LLM 看整 plan + 整 trace，產 整份新 plan

失敗模式：reflect 改 A 搖到 B，issue count 容易升（觀察：2 → 4 → 4 → 放棄）

0–2整 plan 反思次數/build
~60k單次 reflect tokens (含 catalog)
1/33x harness 通過率

v8 (proposed)

反思時機：每跑完 1 個 op 就 check exec_trace（已就位的 auto-preview）

反思 scope：LLM 只看 1 個 op + 前 2-3 個 upstream trace，產 單一 op patch

保留：finalize-time inspect + reflect_plan 當 safety net，處理 emergent issues

0–Nper-op 反思次數/build（N≈ops 數）
~1.5k單次 reflect tokens (局部 trace)
≥2/3 目標3x harness 預期通過率

2. v7 完整 flow — step-by-step

2.1 Topology

START │ ▼ plan ─────────────────────────────────────► validate │ errors? ─────────┤ yes ↓ no ↓ │ repair_plan route_after_validate │ │ └──────► validate │ is_from_scratch + !skip_confirm? │ yes ─► confirm_gate │ ▼ canvas_reset │ ▼ ┌─────► dispatch_op ◄─────┐ │ │ │ │ ▼ │ │ call_tool │ │ │ │ │ status=error? │ │ yes ↓ no ↓ │ │ repair_op cursor < len? │ │ yes ↑ no ↓ │ └────┘ finalize │ │ │ ▼ ◄── 整 plan 跑完才看 │ inspect_execution │ │ │ 有 issues + budget? │ yes ↓ no ↓ │ reflect_plan layout │ │ │ │ │ ▼ │ └─► validate (loop 整套) │ │ └───────────────── ▼ END

2.2 Step-by-step

Step / Node 角色 這一步做什麼 State 變化 / 重要欄位
① plan_node MODEL 讀 instruction + block catalog (52 blocks)，產出 plan = list[Op] (add_node / connect / set_param / run_preview / remove_node)。輸出還含 plan_summary + expected_outputs。LLM 一次看整 catalog (~55k tokens system prompt)。 set plan, is_from_scratch, expected_outputs, summary
② validate_plan_node AGENT 對 plan run PipelineValidator (C1-C15 rules: schema、port、orphan、source-less)。沒 LLM，pure compute。errors 用 ErrorEnvelope 結構化（v7 已加 param / given / expected / rationale）。 set plan_validation_errors; 若有 errors → route 到 repair_plan
③ repair_plan_node (if errors) MODEL LLM 看「原 plan + errors」改一版新 plan。不看 catalog，只靠 errors message。bounded by MAX_PLAN_REPAIR=3。 set new plan, increment plan_repair_attempts
④ confirm_gate (interrupt) AGENT FROM_SCRATCH + 非 skip_confirm 時，graph interrupt() 暫停，發 confirm_pending SSE，等 frontend 回 POST /agent/build/confirm 才繼續。 set user_confirmed
⑤ canvas_reset AGENT FROM_SCRATCH 時清掉 base_pipeline 殘留節點，從乾淨 canvas 開始。 set final_pipeline = empty PipelineJSON
⑥ dispatch_op AGENT 讀 plan[cursor]，包裝成 BuilderToolset 的 args。logical → real id 解析 (n1 → 真實 db id)。 不 mutate state，只 hand args off to call_tool
⑦ call_tool AGENT 呼叫 BuilderToolset.{add_node, connect, set_param, …}。成功則更新 pipeline；失敗 raise ToolError。v7 已加：auto-preview 在 add_node (source-cat) / connect / set_param 後跑一次 toolset.preview(node_id)，寫入 state.exec_trace[lid] = {rows, cols, sample, error}。 append to final_pipeline, set plan[cursor].result_status, write exec_trace[lid], advance cursor
⑧ repair_op_node (if ToolError) MODEL LLM 看當前 op 的 ToolError envelope，產修正 params。只處理 schema-level 錯誤（required missing / type wrong / port mismatch），不處理 runtime 資料問題。bounded by MAX_OP_REPAIR=2。 set plan[cursor].params, increment repair_attempts
↻ loop ⑥-⑦-(⑧) AGENT cursor 沒到 plan 結尾就回 dispatch_op。關鍵：v7 在這個 loop 內不看 exec_trace 是否健康。即使 op 跑完 rows=0，也照樣往下一個 op 走。 cursor 逐步前進到 len(plan)
⑨ finalize_node AGENT (a) Run PipelineValidator on final pipeline → 收 structural_issues (orphan / source-less / missing required)。(b) 若 status=finished，run dry_run via executor.execute(pipeline, inputs=state.trigger_payload) → dry_run_results。inputs 走 trigger_alias 展開（equipment_id↔tool_id, step_id↔step）。 set status, structural_issues, dry_run_results
⑩ inspect_execution AGENT 掃 dry_run_results.node_results 找 3 種 signal：(a) single_point_chart (chart 的 distinct eventTime < 3)；(b) DATA_EMPTY (rows == 0)；(c) node_execution_failed (status != success)。也讀 state.structural_issues 一併彙整。 set inspection_issues (list of ErrorEnvelope dicts)
⑪ reflect_plan_node (if issues + budget) MODEL LLM 看整 plan + 完整 NODE TRACE + 所有 issues + 整 block catalog，重出整份 plan。system prompt 是 plan_node 的 system + reflection prologue (~55k tokens)。bounded by MAX_REFLECT=2。 set new plan, reset cursor=0, logical_to_real={}, final_pipeline=None, exec_trace={}, set skip_confirm=True (避免重 confirm)，status=running
↻ loop ②-⑦-⑨-⑩-⑪ AGENT reflect_plan → validate → canvas_reset → dispatch_op 從頭再跑一遍 patched plan。每一輪整 plan 重來，舊 exec_trace 全丟。 —
⑫ layout_node → END AGENT 對最終 pipeline 跑 dagre 自動佈版，set position.x/y 給 React Flow 渲染。發 SSE done。 set final_pipeline.nodes[*].position

2.3 v7 的痛點

  • 反思太晚：op 跑完一路推到 finalize，前面 op 的 rows=0 / column 錯沒人攔。當 inspect 在 finalize-time 才看，整 plan 已 corrupted。
  • delta 太大：一次重寫整份 plan，LLM 修一處戳一處。觀察：issue 數 2 → 4 → 4 → 放棄 partial fix ship。
  • LLM 看得太多：catalog 55k tokens 進 system，trace 還在 user msg，token budget 很容易爆，response 容易截斷。
  • 無觀察性：「LLM 一次重寫」是黑盒，看不到它修哪、為什麼選那個 node 改。

3. v8 完整 flow — step-by-step（per-op reflection）

3.1 Topology

START │ ▼ plan ─────────────────────────────────► validate │ (… repair_plan / confirm / canvas_reset 跟 v7 一樣) │ ▼ ┌─────► dispatch_op ◄────────────┐ │ │ │ │ ▼ │ │ call_tool │ │ │ │ │ (op 跑完，exec_trace 已寫) │ │ │ │ │ ToolError raise? ─yes─► repair_op │ │ no │ │ ▼ │ │ ★ NEW: check exec_trace │ │ [just_touched_node] │ │ │ │ │ rows=0 / error? │ │ yes ↓ no ↓ │ │ ★ reflect_op cursor < len? │ │ │ yes↑ no ↓ │ │ │ │ finalize │ │ patch │ │ │ │ params │ ▼ │ │ │ │ inspect_execution│ │ cursor-- │ │ │ │ │ │ 有 emergent issue?│ │ └───────┘ │ │ │ yes↓ no↓ │ │ reflect_plan layout │ │ │ │ │ └──────────────────┘ ▼ │ END │ │ ★ = 新節點 / 新判定 │

3.2 Step-by-step（只列跟 v7 不同的部分；其他同 v7）

Step / Node 角色 這一步做什麼 State 變化
① – ⑧ 同 v7 AGENT plan / validate / repair_plan / confirm_gate / canvas_reset / dispatch_op / call_tool / repair_op 全部不變。call_tool 完成後 exec_trace[just_touched_node] 已被寫入（v7 已 ship 的 auto-preview）。 —
★ ⑦.5 _detect_op_issue AGENT 純函數，無 LLM。call_tool 成功後立刻 check：
• exec_trace[lid].error 是不是 not None（executor 噴錯）
• exec_trace[lid].rows == 0 而且 cursor >= 3（避開前 1-2 個 source 還沒接 input 的正常 0 row）
• (預留) chart_spec missing 必要 field
有 issue → set state.last_op_issue (ErrorEnvelope dict)；無 → fallthrough 到 route_after_call。 		set last_op_issue (envelope dict) or None
★ ⑦.6 route_after_call (改寫) AGENT 新路由優先序：
1. op error → repair_op (既有)
2. last_op_issue is not None + reflect_op_attempts[lid] < MAX_REFLECT_OP=2 → reflect_op
3. cursor < len → dispatch_op
4. done → finalize 		—
★ ⑦.7 reflect_op_node MODEL 小 scope LLM call。system prompt ~800 tokens（不含 catalog）。user message 只含：
• USER PROMPT (instruction[:400])
• FAILING OP (current op 的 dict)
• UPSTREAM TRACE 前 2-3 個 op 的 NODE TRACE
• ISSUE envelope (code/param/given/expected/rationale)
• 這個 op 的 block param_schema
LLM 輸出 JSON：{action:"patch_params", new_params:{...}} 或 {action:"rollback", rollback_to_cursor:K, new_params_for_K:{...}}。 		視 action 而定：
• patch: 改 plan[cursor-1].params，cursor--
• rollback: 改 plan[K].params，cursor=K，清 exec_trace[K..] 跟 plan[K+1..].result_status
increment reflect_op_attempts[lid]
★ ⑦.8 re-dispatch AGENT cursor 已倒退 → graph edge reflect_op → call_tool 重跑剛才那個 op (帶新 params)。重新 preview 寫 exec_trace。再過一次 _detect_op_issue：
• 好了 → 繼續下一個 op
• 還壞 → 再 reflect_op (但 attempts 已 +1，2 次後 ban，帶傷繼續) 		—
⑨ – ⑪ 同 v7 (保留 safety net) AGENT finalize / inspect_execution / reflect_plan 全保留，處理 emergent issue：single_point_chart 這種「整 chain 串起來才看得出來」的問題，per-op 抓不到，留到 finalize-time 抓。多數 case 在 ⑦.7 已修掉，這層觸發機率降低。 —

3.3 v8 的設計賭注

  • 小 delta 風險低：一次只改 1 個 op 的 params，LLM 不會「修 A 戳 B」。最壞 case 是該 op 永遠修不好，但不會 cascade。
  • 反思時機正確：問題發生當下就 reflect，不等到 plan 走完才回頭。前面 op 修好，後面 op 才有正確 input。
  • cost 整體可能更低：1 次 reflect_plan ≈ 60k tokens (整 catalog)；1 次 reflect_op ≈ 1.5k tokens。即使 reflect_op fire 20 次也比 reflect_plan 1 次便宜。
  • 觀察性好：每次 reflect_op SSE 帶 op cursor + 修改 reason，frontend 可以顯示「正在修第 N 個 op...」。

4. 具體 scenario walkthrough

User prompt：「檢查機台 $equipment_id 最後一次 OOC 時，是否有 >2 張 SPC chart 也 OOC，並顯示該 SPC charts」

假設 LLM 出的初版 plan：

n1: block_process_history(tool_id=$equipment_id, limit=1, nested=true)   ← ⚠ limit=1 太小
n2: block_unnest(column='spc_charts')
n3: block_filter(column='is_ooc', operator='==', value=true)
n4: block_line_chart(x='eventTime', y='value', facet='name')
n5: block_step_check(column='spc_summary.ooc_count', operator='>=', threshold=2)
edges: n1→n2, n2→n3, n3→n4, n1→n5

真正的 bug：n1 設了 limit=1 → 下游 chart 一定 single-point。

4.1 v7 怎麼處理這個 case

時點node發生什麼
t=0plan_nodeLLM 出上面那 5 個 op + 4 個 edge。
t=1validate沒結構錯誤，schema 也 OK。pass。
t=2dispatch+call_tool ×5 個 add_node + 4 個 connect 一路推。exec_trace 寫到 n1 rows=1 (因 limit=1)；n2 rows=3 (1 row × 3 spc_charts)；n3 rows=1 (filter is_ooc=true 留 1)；n4 chart 收到 1 row。但沒人看 exec_trace 是否健康。cursor 推到 9。
t=3finalizevalidator 過。dry_run 跑完，dry_run_results 顯示：n4 chart 只有 1 distinct eventTime；n5 step_check value=0。
t=4inspect_executionsingle_point_chart 命中 → set 1 issue：「node 'n4' resolved to only 1 distinct eventTime」。
t=5reflect_plan #1LLM 看整 plan + 整 trace + 整 catalog (~60k tokens)。出新 plan，但它可能把 n1 改成 limit=100，也可能改 n3 filter 條件，也可能完全換 block。output 16 ops。
t=6validate (new plan)常見：新 plan 出現 orphan node 或 column 引用錯 → 17 hard errors。
t=7repair_plan ×3連續 3 次修，都還有 errors → plan_unfixable 路徑 → finalize fails 0 nodes built → END。
t=8結局canvas 0 nodes 或 partial fix。harness 失敗。原本只是 n1.limit=1 一個小問題，因為整 plan 重寫被搞糟。

6. 失敗的可能 + 對策

風險偵測對策
reflect_op LLM 不接受結構化輸出（rollback / patch_params）JSON parse fail rate > 30%強化 system prompt 範例 + 退回到只支援 patch_params (拿掉 rollback) 看穩定度
每個 op 都觸發 reflect_op，build 變超慢build 時間 >10 min提高 _detect_op_issue 觸發條件嚴格度（e.g. rows=0 才算）
reflect_op 永遠卡 budget 用完reflect_op_attempts 都到 2review prompt + 提供 block description 是否 sufficient
rollback 把好的 op 也搞壞rollback 後新的 _detect_op_issue 又 fire限制 rollback distance ≤ 3，超過用 patch_params only
跟 reflect_plan 衝突finalize-time 還是有 single_point_chart正常 — 那是 emergent issue，留給 reflect_plan 處理

7. 觀察指標 (KPI)

  • 主要：harness 3x 通過率從 1/3 升到 ≥ 2/3（理想 3/3）
  • 次要：finalize-time reflect_plan 觸發次數降低（per-op 先處理掉）
  • cost：總 LLM tokens/build 不超過 v7 的 1.5×
  • 觀察性：每次 reflect_op SSE 帶 cursor + reason，sidecar log 可看到「reflect_op#1 cursor=3 action=patch_params reason=…」

8. Out of Scope

  • chart_spec missing field 偵測（留 v8.1，先處理 rows=0 / executor error 兩個基礎 signal）
  • rollback distance > 3（複雜場景，等 v8 stabilize 再放鬆）
  • streaming reflect_op SSE 給 frontend 顯示「fixing op#N…」UX（v9 改 UI）
  • reflect_op 跨 build 學習 / cache（規模太小，不值得）
