02兩代架構：單代理 vs 多代理

多代理 Phase 0（2026-07-02）把單一 builder 迴圈拆成 Planner / Builder / Repair 角色代理 + Verifier（無 LLM 確定性閘門）+ Supervisor（事後鑑識）。改造目標不是跑分，是可觀測與可治理。

05GLM 供應端風暴（07-05 ～ 07-06）

07-06 兩個測試案「卡住」的真因。短 prompt 探測 8/8 正常、0 個 HTTP 錯誤 — 只有長 context 建置 prompt 觸發空回應（finish=stop、零內容），指向 Fireworks 端高負載降級。

時段空回應率影響
正常期（06-25 ～ 07-04）低（17/17 可重現）—
07-05 早上間歇（9+3 次 escalate）零星 handover
07-06 下午～晚間16–18%（區間峰值 ~30%）簡單 xbar 案也 handover；17 案跑不完
已有防線：空回應率日統計（S3 rollup）+ 連 3 空 escalate 守門。 待議：空回應率超標自動 fallback 到 Anthropic 的保險絲。

06建議

層級配置理由
Default（建議）Sonnet 5 + adaptive thinking(high)，直連 唯一「首輪 17/17 + 最快 + DAG 最精簡」三者兼得；$2.23/17 案（單案約 $0.13）在互動建置情境可接受
成本優先備選Haiku 4.5 快答，直連$1.02 便宜一半、同速；偶發 flake 由既有 C16 閘門攔下重試即可
備援GLM-5.2（OpenRouter/Fireworks）正常期 17/17 且最便宜；受供應端風暴影響，降為備援並持續監測空回應率
退役KIMI K2.5 / Sonnet 走 OpenRouter前者品質落後；後者 cache 失效使成本假性膨脹 5 倍
