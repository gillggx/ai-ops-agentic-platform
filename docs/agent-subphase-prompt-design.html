Agent Sub-Phase Prompt Design

Glass Box Builder — context-aware prompt assembly per agent action · 2026-06-12 · gated behind ENABLE_RICH_CANVAS_SNAPSHOT

TL;DR. Glass Box agent 跑在一個 sub-phase 狀態機 (pick → construct → tune)。 現在 prompt 只分「phase 第 1 round 給 rich context / 第 2+ round 給 light diff」， 跟 agent 當下在做什麼無關。結果：填 param 的 round 看不到 upstream node 的 output columns，agent 只能盲猜欄位名 — SLASH-13 的 box_plot 把 recipe 猜成 step 就是這樣來的。本文件定義「每個 sub-phase 該給什麼 context」的 matrix， 作為重構依據。

1. Agent 行為狀態機

phase entry → pick → construct → tune → verify → phase advance / chain next block

每個 sub-phase 只開放特定 tool（graph 層 deterministic filter，不靠 prompt 自律）。 轉移由 _TRANSITIONS 純函數決定。

Sub-phase開放的 toolsAgent 做的決定轉出條件
pick commit_pick, inspect_node_output, inspect_block_doc, abort_phase (+add_node 若 auto_signal) 下一個要加哪個 block commit_pick → construct
construct add_node, connect, abort_node, inspect_node_output, inspect_block_doc add_node 填 params + connect 到正確 upstream connect → tune
tune set_param, run_verifier, commit_pick, abort_node, inspect_node_output 修 param / 接下一個 block / 觸發驗證 run_verifier → verify · commit_pick → construct
verify deterministic node — 非 LLM 結構檢查 terminal 是否符合 phase.expected ADVANCED → 下一 phase / REJECTED → tune

2. 現況 — prompt 是「兩段式」不是「按行為」

時機Builder內容含 columns?
phase 第 1 round _build_observation_md pipeline inputs + exec_trace runtime schema + connect options + action history YES
phase 第 2+ round
(絕大多數決策點)		 _build_canvas_diff_md verifier feedback + 裸 canvas (id / block_id / params) + connect options + goal NO
三個結構性問題：
  • column-blindness — 第 2+ round 的 canvas 完全沒有 node output columns / sample。 填 param 時無從得知 upstream 有哪些欄位。
  • sub-phase hint 只出現一次 — 黏在第 1 round 的 seed，之後不再重印； agent 過幾 round 就忘了自己在 pick / construct / tune 哪個模式。
  • 同一份 diff 給所有 sub-phase — pick 跟 tune 需要的 context 天差地遠， 卻拿到一模一樣的 canvas snapshot。

3. 設計核心 — 行為 → 需要的 context

column-blindness 害到的是 construct + tune（真正填 param 的兩個）。 pick 需要的是 flow 形狀 + 終端資料預覽，不是完整 schema。

4. 目標 context 設計（三張卡）

pick — 看 flow，決定下一步

== SUB-PHASE: pick_block ==   (每 round 重印)

== FLOW SO FAR ==
  [source] n1 block_process_history(EQP-01, APC, 14d)
      └─> n2 block_unnest(APC.parameters)        <- canvas terminal
  terminal n2 cols: [eventTime, toolID, lotID, step, etch_time_offset,
                     chamber_temp, gas_flow, …(+4 more)]

PHASE GOAL: 展示各 Recipe etch_time_offset 分佈 (expected: chart)
You're choosing the NEXT block. Terminal n2 exposes the cols above —
pick a block whose inputs those cols satisfy.
-> commit_pick(block_id) when decided, or inspect_node_output(n2) for full sample.

flow 用樹狀畫（source → ... → terminal），比平鋪 edge 好讀。 終端 node 的 cols 摘要就地給，agent 不用再花 round inspect。

construct — 填 params + 接線

== SUB-PHASE: construct_node ==
You committed to block_box_plot. Now add_node + connect upstream n2.

== UPSTREAM n2 OUTPUT (what you're connecting FROM) ==
  cols: [eventTime, toolID, lotID, step, etch_time_offset, chamber_temp, …]
  sample: {eventTime:'2026-05-29...', toolID:'EQP-01', step:'STEP_004',
           etch_time_offset:3.2, chamber_temp:200, …}
  [note] no `recipe_id` / `recipe` column present — if you need recipe
         grouping, transform first (this is a flat APC table).

== CONNECT OPTIONS ==
  n2.data (dataframe) -> block_box_plot.data  ok

Fill add_node params using ONLY cols listed above.

治本：upstream cols + sample 直接塞到 agent 填 param 的當下。 加「你要用的 column 不在 upstream」的 deterministic 警告（能精確 parse 時才報）。

tune — 修哪裡

== SUB-PHASE: tune ==
== VERIFIER FEEDBACK == (if rejected)
  rejected: block_box_plot — x='step' but goal wants Recipe-level grouping
== NODE n3 block_box_plot CURRENT params ==
  x='step'   y='etch_time_offset'   title='各 Step…'
== UPSTREAM n2 cols (valid values for x / y) ==
  [eventTime, toolID, step, etch_time_offset, …]   (no recipe_id!)
-> set_param to fix, or commit_pick to chain a transform that produces recipe_id.

5. 案例 — SLASH-13 為何失敗（column-blindness 實證）

User prompt：「EQP-01 過去 14 天每個 recipe 的 APC etch_time_offset 分佈 box plot 對比」

Agent 在 chart phase round 10 (call 23) 看到的完整 prompt：

== CANVAS NOW (2 nodes, 1 edges) ==
  n1 [block_process_history]  params={tool_id='EQP-01', object_name='APC', time_range='14d'}
  n2 [block_unnest]  params={column='APC.parameters'}
  edge n1->n2
PHASE GOAL: 展示各 Recipe 蝕刻時間偏移量的分佈對比 (expected: chart)
沒有 n2 的 output columns。 Agent 只能猜，於是 emit： 
add_node block_box_plot(x='step', y='etch_time_offset', title='各 Step 蝕刻時間偏移量分佈對比')
把 recipe 猜成 step（連 title 都寫錯）。它自己 commit_pick 的 reasoning 寫： "the most relevant categorical field is 'step' (which represents different process steps/recipes)" — 純 guess，且把 step 跟 recipe 混為一談。 verifier 連 reject 2 次後 round budget 耗盡 → handover。
修法後（construct context 給 cols）： agent 會在 prompt 直接看到 n2 cols: [..., step, etch_time_offset, ...] 且 沒有 recipe_id — 於是 realize 需要先 transform 拿到 recipe 維度，主動補 block_select / 改用 flat-mode source， 而不是盲填 x='step'。

7. 風險控制

風險處理
node 未 preview（exec_trace 無 key）只印 id/block/params + 標 (not yet previewed)
寬表 50+ colscap 30 + …(+N more)；完整清單要 inspect_node_output
sample 深 nested第 1 層展平，深層 value 縮 {...} / [...]
token 暴增flag 控 + 每 node cols≤30 / sample≤600 char。預估 +0.5–1.5 KB/round，遠少於省下的 5–10 round
「column 不存在」誤報只在能精確 parse agent 上一個 tool_args column refs 時才警告；parse 不出就不報
破壞既有過關 caseflag off 完全等同現狀；on 才改。全 17 case 重跑驗證
