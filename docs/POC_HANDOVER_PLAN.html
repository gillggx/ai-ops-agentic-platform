接手順序與方式（三階段）

階段 1核心 · 先接

LLM Agent + Knowledge（python_ai_sidecar）

平台價值與最高複雜度都在這。其餘一切只是「怎麼驅動它」或「怎麼維運它」—— 所以一定先接這層，其它層才有依託。

範圍
  • Agent 三條 stack：Chat orchestrator_v2、Glass Box builder（goal_plan → agentic_phase_loop → phase_verifier → finalize）、Block advisor
  • LLM client：provider 切換（KIMI/OpenRouter）、bounded retry、finish_reason 觀測
  • Knowledge：agent_knowledge（plan/execute hints）、block_docs（描述＝唯一真相）、pgvector RAG
  • Block catalog ＋ 56 個 executors
接手方式
  • 讀 CLAUDE.md 的 Agent 原則：flow 寫 graph 不寫 prompt、禁 case-rule、description 是唯一真相
  • 跑回歸套件當「會動」門檻：tools/slash17/（builder）、tools/chat_eval/（chat）
  • 除錯：Build Trace（/tmp/builder-traces）＋ tools/trace_replay/
  • 掌握 env 開關：LLM provider（sidecar .env OLLAMA_MODEL）、feature_flags
入門資料

CLAUDE.md · docs/history/PROJECT_HANDOFF.md · tools/slash17/BASELINE.md

接手完成判準：能跑 SLASH-17 + chat_eval 全綠 · 能讀一份 build trace 並解釋失敗在哪 · 能安全切換 LLM provider 與 feature flag。
階段 2介面 / 設定面

Pipeline Builder · System MCP · Skill 庫（須保留使用者設定）

這層是使用者「產生價值 + 存設定」的地方。理解核心後接這層；重點是 設定狀態必須完整保留（skills、MCP、blocks 都在 DB）。

範圍
  • Pipeline Builder（Glass Box 自動 + 手動）— 前端 UI ＋ 它驅動的 build graph
  • System MCP admin：外部資料設定 ＋ ${ENV} 機密插值
  • Skill 庫 L1/L2/L3（瀏覽 / 編寫 / Try-Run）、Block Docs drawer
接手方式
  • 保留 DB 狀態：pb_blocks / skills / mcp_definitions 不可遺失
  • 改 block 一定走「描述唯一真相」：description + param_schema + examples 同步改 ＋ DB 同步（見 new-block checklist）
  • 前端只 proxy，不放業務邏輯；設定都落 Java/DB
入門資料

POC_DEPLOYMENT_SPEC.html §4（System MCP）· POC_PRODUCT_SPEC.html（功能規格）· 各 block 的 description · new-block 5-點 checklist

接手完成判準：能新增/修改一個 block（5 點 + DB 同步）而不破壞開機 block-consistency（56/56/56/56）· 能端到端設定一個 System MCP 並 Try-Run 出結果。
階段 3周邊 · 依序

Auth / 觀測 / 部署 / 基礎設施

重要但可先靠 runbook 操作、邊學邊接。建議內部順序：Auth → 觀測 → 部署/DB → K8s。

子項內容方式
Auth / RolesJWT、OIDC（選用）、role hierarchy IT_ADMIN > PE > ON_DUTY、role audit讀 OIDC envs；測試帳號驗 role gating
觀測 / TraceBuild Trace、boot invariants、journal logs把 SLASH-17/chat_eval 接進 CI 當 merge gate
部署 / DBEC2 systemd（deploy/*.sh）、Flyway prod 關閉→手動 psql、canonical block seed從零 setup.sh 一遍；手動套一支 V*.sql
基礎設施EC2 → K8s（manifests 已備、run wrapper 待補）等 K8s target env 決定後補 run.sh
接手完成判準：能在乾淨環境從零部署（setup.sh）並手動套 migration · 完整擁有 auth/roles。

接下來的重點（Roadmap）

主題重點狀態
準確度守住 SLASH-17 ~17/17；完成 chat-mode eval（B 模式：確認後真的建+跑）；knowledge 層補 domain model（process-centric）SLASH-17 baseline 已建；chat eval harness 已建
Robustness補開機 block-consistency 觀測洞；regression driver 加 wall-clock cap（避免 server stall 卡死）已知，待補
成本KIMI K2.5 為 cost-right default；prompt cache；逐 stage 成本盯單一 switch = sidecar .env
待決策（產品）goal_aware on/off（需 3×median 重評）；ON_DUTY chat 數據問題 UX（目前會導去建 pipeline → 值班帳號建不了）待 stakeholder 拍板
工程化CI 串回歸套件當 merge gate；K8s manifests 落地；DB migration 流程文件化下一階段

建議的起手式（第一週）

  1. 讀 CLAUDE.md + POC_PRODUCT_SPEC.html（功能規格），建立心智模型（架構邊界 + 原則）。
  2. 本地/測試環境部署 POC branch，跑通 §10 acceptance smoke。
  3. 跑一次 tools/slash17/run.sh + grade_strict.py 與 tools/chat_eval/run.sh， 對齊 BASELINE.md —— 這是「平台健康」的客觀基準。
  4. 挑一個 SLASH-17 case，從 build trace 走一遍 plan → 每步，學會除錯語言。
  5. 做一次「無害改動」練習：改一個 block 的 description（走 5 點 + DB 同步），確認 boot invariant 不破。
