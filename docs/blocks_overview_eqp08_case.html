EQP-08 OOC 案例 — 用到的 7 個 Block

以 user prompt：「EQP-08 機台最後一次 OOC 事件時刻，統計並顯示有多少張 SPC charts 也同時 OOC，篩選 OOC 數量 > 2 的情況，以 SPC Panel 分別列出各 chart 的數值、UCL/LCL 及 OOC 狀態」為例。 Output sample 用 EQP-01 STEP_001 真實資料 (12 SPC charts, r_chart is_ooc=true)。

1. process_history 2. unnest 3. filter 4. groupby_agg 5. sort 6. step_check 7. spc_panel 📊 兩種解法對照

block_process_historySOURCE

從 ontology 拉指定條件 (tool_id / lot_id / step) 的 process events。每筆 = 一次 process，預設帶 nested SPC/APC/DC/RECIPE/FDC/EC sub-objects + 預算 spc_summary。

何時用
  • 已知 tool_id / lot_id / step，要查 process 歷史做分析
  • 例：「EQP-01 最近 50 次 SPC trend」「LOT-123 在 STEP_004 的 APC 參數」
  • ❌ 「現在哪些機台在跑」→ 用 block_list_objects

INPUT

無 input port (source block)

PARAMS

nametypedefault
tool_idstring三擇一
lot_idstring三擇一
stepstring三擇一
object_nameenum"" (全部)
time_rangestring"24h"
limitint100
nestedbooltrue

OUTPUT (real sample, EQP-01 STEP_001)

port: data (dataframe, nested) 
{
  "eventTime": "2026-05-15T08:36:31.166000",
  "toolID": "EQP-01",
  "lotID": "LOT-0541",
  "step": "STEP_001",
  "spc_status": "OOC",          ← rollup
  "fdc_classification": null,
  "spc_charts": [               ← list[12]
    {"name":"xbar_chart","value":16.6,"ucl":17.5,"lcl":12.5,"is_ooc":false,"status":"PASS"},
    {"name":"r_chart",   "value":905.7,"ucl":880,"lcl":820,"is_ooc":true,"status":"OOC"},
    {"name":"s_chart",...}, ...12 charts
  ],
  "spc_summary": {              ← precomputed!
    "ooc_count": 1,
    "total_charts": 12,
    "ooc_chart_names": ["r_chart"]
  }
}
🟢 OOC 4 個欄位（同件事不同粒度，選一個就夠）：
spc_status == "OOC" — 整 event 有沒有任一 chart OOC (rollup)
spc_summary.ooc_count — 這 event 有幾張 chart OOC (int)
spc_summary.ooc_chart_names — 哪幾張 OOC (list[str])
spc_charts[i].is_ooc — 第 i 張 chart OOC 與否 (要先 unnest)
⚠ 三擇一 (tool_id/lot_id/step) 都不填 → NO_FILTER_GIVEN
⚠ time_range "過去 2 天" 要寫 "48h" 不是 "1d"

block_unnestTRANSFORM

把 list 欄位 explode 成多筆 rows，object element 的 keys 自動 lift 成 top-level columns（pandas explode）。

何時用
  • 「想對單一 SPC chart 做趨勢」→ unnest spc_charts，再 filter name='xbar_chart'
  • 「想看 APC 各參數獨立 series」→ unnest APC.parameters
  • ❌ 「想算 OOC 數量」→ 不要 unnest，用 spc_summary.ooc_count 即可

INPUT

port: data (dataframe，含至少 1 個 list 欄位)

PARAMS

nametype
★ columnstring

e.g. 'spc_charts' / 'APC.parameters'

OUTPUT (unnest 'spc_charts' on above sample)

port: data — 12 rows (原 1 row × 12 charts) 
row 0: {eventTime, toolID, lotID, step, spc_status,
        name:"xbar_chart", value:16.6, ucl:17.5,
        lcl:12.5, is_ooc:false, status:"PASS"}
row 1: {..., name:"r_chart", value:905.7,
        is_ooc:true, status:"OOC"}
row 2: {..., name:"s_chart", ...}
...
row 11: {..., name:"match_tune_chart", ...}

cols: [eventTime, toolID, lotID, step, spc_status,
       fdc_classification, spc_summary,
       name, value, ucl, lcl, is_ooc, status]
(原 spc_charts 欄消失，6 leaves 變 top-level)
⚠ leaf 欄位實名是 name / value，不是 spc_name / chart_name

block_filterTRANSFORM

單條件過濾：保留 column <operator> value 為 true 的 rows。

INPUT

port: data (dataframe)

PARAMS

nametype
★ columnstring
★ operatorenum
valueany

operator: ==, !=, >, <, >=, <=, in, contains

OUTPUT (filter is_ooc==true on n2 output)

port: data — 12 rows → 1 row 
row 0: {
  eventTime: "2026-05-15T08:36:31.166000",
  toolID: "EQP-01",
  name: "r_chart",
  value: 905.75,
  ucl: 880.0,
  lcl: 820.0,
  is_ooc: true,
  status: "OOC"
}

cols 不變 (沒新增/刪除)
⚠ column 必須是上游真實欄位 (case-sensitive)
⚠ in 的 value 必須是 list；contains 只對 string 欄位
⚠ value 是 literal — 不是 placeholder 像 $X 或 {last_time}。要跨 branch 取值請用 block_join

block_groupby_aggTRANSFORM

分組 + 單一聚合函數 (pandas groupby + agg)。Output column 名是 <agg_column>_<agg_func>。

INPUT

port: data

PARAMS

nametype
★ group_bystring
or list
★ agg_columnstring
★ agg_funcenum

agg_func: mean, sum, count, min, max, median, std

OUTPUT (groupby eventTime + count(name))

port: data 
row 0: {
  eventTime: "2026-05-15T08:36:31.166000",
  name_count: 1   ← 1 張 chart 在這時刻 OOC
}

cols: [eventTime, name_count]
(輸出 column 是 name_count，不是 'count'!)
⚠ 下游 sort/chart 引用要寫 name_count 不是 count
⚠ 多欄 group_by 用 list ['toolID','step'] 不是逗號分隔字串

block_sortTRANSFORM

多欄排序 + optional top-N (head(N))。常用於 ranking / 取最後 N 筆。

INPUT

port: data

PARAMS

nametype
★ columnsarray
limitint

columns = [{column:'X', order:'asc'|'desc'}]

OUTPUT (sort eventTime desc, limit=1)

port: data 
row 0: {
  eventTime: "2026-05-15T08:36:31.166000",
  name_count: 1
}

cols 不變，只保留最新 1 筆 (last OOC time)
⚠ columns 是 list of objects，不是 list of strings
⚠ limit 是 head(N)，要取最大值請先 desc 再 limit

block_step_checkCHECK / TERMINAL

Skill terminal — 聚合上游 + 與 threshold 比對，輸出 {pass, value, ...}。SkillRunner 從這讀 pass/fail。

何時用
  • ✅ Skill step 結尾的 pass/fail check
  • ✅ 量化比較：count / sum / mean / max / min / last / exists
  • ❌ 終端 block — 不要接下游 (especially 不要接 block_alert)

INPUT

port: data (dataframe，行數可變)

PARAMS

nametypedefault
aggregateenum"count"
columnstring非 count/exists 必填
★ operatorenum">="
thresholdnumber數值比較必填

OUTPUT (count, >=, threshold=2 on n5)

port: check (dataframe，1 row) 
row 0: {
  pass: false,           ← 1 不 >= 2
  value: 1,
  threshold: 2,
  operator: ">=",
  aggregate: "count",
  column: null,
  note: "count=1 not >= 2",
  evidence_rows: 0
}
💡 OOC count 捷徑（推薦） — 跳過 unnest+filter+groupby_agg+sort，直接：
step_check(column='spc_summary.ooc_count', operator='>=', threshold=2)
接在 block_process_history 後面即可，2 個 node 完成。

block_spc_panelCOMPOSITE

Parameterized SPC chart composite — 給 step + chart_name 兩必填，內部撈 process_history → unnest → 出 line/bar chart + UCL/LCL + OOC highlight。1 個 node 全包。

何時用
  • ✅ user 說「看 SPC chart」「SPC Panel」「機台最後 OOC 時的 SPC 狀況」
  • ✅ 取代 4 步 unnest+filter+sort+line_chart 組合
  • ❌ 計算 Cpk/EWMA → 用統計類 block

INPUT (transform mode 才用，否則自己 fetch)

port: data (opt) — process_history nested

PARAMS

nametypedefault
★ stepstring
★ chart_namestring
tool_idstring(留空=跨機台)
time_rangestring"7d"
event_filterenum"latest_ooc"
color_bystringe.g. toolID

event_filter: latest_ooc / latest_event / all / custom_time

OUTPUT (xbar_chart, event_filter=all)

port: chart_spec (dict) 
{
  "type": "line",
  "title": "SPC Charts",
  "x": "eventTime",
  "y": "value",
  "data": [
    {"eventTime":"05-15T08:36","value":16.6,"name":"xbar_chart"},
    {"eventTime":"05-15T07:22","value":15.8,"name":"xbar_chart"},
    ... N 個時間點
  ],
  "rules": [
    {"value":17.5,"label":"UCL","color":"#f59e0b"},
    {"value":12.5,"label":"LCL","color":"#f59e0b"}
  ],
  "highlight": {"field":"is_ooc","color":"red"},
  "meta": {"event_filter":"all","n_rows":50,"n_series":1}
}
⚠ step + chart_name 都必填（即使 transform mode 也是）
⚠ 不要在 spc_panel 前接 unnest/filter/sort — composite 自己處理，pre-filter 會砍剩 1 row
⚠ chart_name 是完整 key (xbar_chart) 不是裸 xbar

📊 同一個 user prompt — 兩種解法對照

關鍵差異：verdict branch 用 spc_summary.ooc_count 直讀，還是 unnest+filter+count 自己算。
Description 已有提示但被埋深 → LLM 90% 選錯走長路 → chain 越長越多失敗點。

❌ 長路徑 (現在 macro_plan 常選的)

7 nodes, verdict branch 5 步

1. block_process_history (tool_id=EQP-08)
2. ↓ block_unnest (spc_charts) → 12N rows
3. ↓ block_filter (is_ooc==true) → ~N rows
4. ↓ block_groupby_agg (eventTime, count) → eventTime+count
5. ↓ block_sort (eventTime desc, limit=1) → 1 row
6. ↓ block_step_check (count_>=, 2)

+ parallel
7. block_spc_panel (step, chart_name, latest_ooc) ← 自己撈

失敗點：n4 column 名 (name_count vs count)、n5 限制太嚴、cross-branch ref 誘惑、phantom edges

✅ 短路徑 (用 spc_summary 捷徑)

3 nodes, verdict branch 1 步

1. block_process_history (tool_id=EQP-08, nested=true)
2. ↓ block_step_check (column='spc_summary.ooc_count', op=>=, threshold=2)

+ parallel
3. block_spc_panel (step, chart_name, latest_ooc) ← 自己撈

好處：spc_summary.ooc_count 由 process_history 預算好；step_check 用 path 文法直讀；無 cross-branch 風險

結論：description 已有 spc_summary.ooc_count 提示但深埋。需要把 「OOC 判定 4 欄位對照表」提升到 block_process_history description 開頭最 prominent 處， 並補一個 「process_history → step_check spc_summary.ooc_count」shortcut example 在 block_step_check。 spc_panel 「典型 pipeline (E)」也要從 5 步 verdict 改成 1 步。
