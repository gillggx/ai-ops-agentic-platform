## Refined Implementation Approach

本專案將在原有 FastAPI 架構基礎上持續推進，採用以下精煉後的實作策略：

**核心框架與函式庫（維持不變）：**
1. **FastAPI** - 主框架，提供自動 OpenAPI 文件生成與高效能非同步處理
2. **SQLAlchemy 2.0 (AsyncSession)** - ORM 框架，支援 SQLite（開發）與 PostgreSQL（生產）
3. **Pydantic v2** - 資料驗證與序列化，含欄位層級驗證規則
4. **Alembic** - 資料庫遷移管理
5. **python-jose + passlib** - JWT 身份驗證與密碼雜湊
6. **uvicorn** - ASGI 伺服器
7. **python-dotenv** - 環境變數管理

**新增實作重點：**
8. **pytest + httpx + pytest-asyncio** - 單元測試與整合測試，目標覆蓋率 80% 以上
9. **logging（Python 內建）** - 結構化日誌記錄，追蹤系統運行狀態
10. **API 版本管理** - 採用 `/api/v1/` URL 前綴進行版本控制

**架構分層（精煉版）：**
- **Router Layer**：掛載於 `/api/v1/` 前綴，包含 auth、users、items 三個子路由模組
- **Service Layer**：封裝業務邏輯，處理權限驗證（僅能修改自己的資源）
- **Repository Layer**：純資料存取，封裝所有 SQLAlchemy 查詢
- **Database Layer**：AsyncSession 非同步連線管理
- **Core Layer**：security（JWT）、exceptions（AppException）、response（StandardResponse）、logging（結構化日誌）

**新增功能模組實作步驟：**
1. 新增 `app/core/logging.py` - 建立結構化日誌系統，記錄請求、回應與錯誤資訊
2. 新增 `app/middleware.py` - 實作請求日誌中介層（Request ID、耗時記錄）
3. 新增 `tests/` 目錄結構 - 涵蓋單元測試（services、repositories）與整合測試（API endpoints）
4. 在 `main.py` 加入健康檢查端點 `/health` 與 API 版本前綴 `/api/v1`
5. 所有回應統一使用 `StandardResponse` 格式，確保一致性
6. 全域異常處理器覆蓋 `AppException`、`RequestValidationError`、`HTTPException` 與通用 `Exception`

## Anything UNCLEAR

目前仍有以下幾點待釐清，但已依最佳實踐進行假設與設計：

1. **requirement.txt 業務規格**：具體的 API 端點清單與業務邏輯細節仍未提供，目前以通用 User + Item 的 CRUD 為基礎實作，若有特定業務領域需求（電商、部落格等）需再調整資料模型。

2. **資料庫環境切換**：已設計支援 SQLite（開發）與 PostgreSQL（生產）透過環境變數 `DATABASE_URL` 動態切換，但 PostgreSQL 的連線池配置細節（pool_size、max_overflow 等）可依實際負載需求調整。

3. **權限控管粒度**：目前設計為「使用者只能修改/刪除自己的資源，超級管理員可操作所有資源」，若需要更細粒度的 RBAC（Role-Based Access Control）機制，需額外實作 roles 資料表與 permissions 管理模組。

4. **Token 刷新機制**：目前僅實作 Access Token，若需要 Refresh Token 機制以延長登入狀態，需在 AuthService 中新增 `refresh_token()` 方法並調整對應的資料庫 schema（儲存 refresh token 黑名單或白名單）。

5. **測試環境隔離**：`tests/conftest.py` 預設使用獨立的 SQLite 記憶體資料庫（`:memory:`）進行測試，確保測試不影響開發資料庫，但若需要針對 PostgreSQL 特定功能進行整合測試，需另外配置測試用 PostgreSQL 實例。

6. **部署配置**：目前尚未提供 Dockerfile 與 Docker Compose 配置，建議後續補充以利團隊開發與 CI/CD 流程整合。
